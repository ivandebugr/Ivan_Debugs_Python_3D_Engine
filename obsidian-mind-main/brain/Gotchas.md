# Gotchas

Things that have bitten before and will bite again.

## View-dependent shader effects need an angle sweep, not one static frame — 2026-07-16
**Context:** v1.7 floor-specular investigation. Verifying a narrow specular highlight (`pow(dot(N,H), 32)`, `lit_shader.py`) in the floor material with a single offscreen camera position.
**Symptom:** A single static frame dump measured the specular term as effectively invisible, producing a false "premise doesn't hold" report — before an angle sweep caught the highlight clearly visible from other yaw/pitch positions.
**Root cause:** Any effect with an N·V or reflection-direction dependency (specular lobes, rim/Fresnel terms) is only bright within a narrow cone of view angles relative to the light and surface normal. A high specular exponent (32 here) narrows that cone further, so one arbitrarily-chosen camera angle can easily land outside it while the effect is fully working.
**Fix:** When verifying any view-dependent shader effect, sweep the viewing angle (yaw/pitch around the surface, not just one shot) as part of the verification — never conclude "not visible" from a single static frame.
**Related:** see the next entry below — this session also hit a second, compounding trap around *where* that verification runs.
**Source:** v1.7 floor-specular investigation, 2026-07-16

## Prefer the live in-game tuning tool over new offscreen harnesses for view-dependent checks — 2026-07-16
**Context:** Same v1.7 floor-specular investigation. The project already has a live in-game tuning tool (`dev_shader_tuning.py`, Ctrl+M) that renders the real game scene and lets uniforms be adjusted while watching the actual view.
**Symptom:** Building new offscreen harnesses to re-verify the same specular behavior surfaced artifacts of the test scene itself, not the real game — happened twice this session (grass-texture noise, a clipped white floor) before the live tool's own in-game check settled the question.
**Root cause:** A purpose-built offscreen harness necessarily constructs its own scene (geometry, camera, lighting) to render headlessly. That scene is not the shipped game scene, so it can introduce its own visual artifacts (texture tiling, clipping, framing) that are easy to mistake for the effect under test — especially for view-dependent effects where framing/geometry choices matter.
**Fix:** For view-dependent tuning (specular, rim/Fresnel, anything camera-angle-dependent), prefer the live in-game tool that reflects the real view. Reach for offscreen harnesses for camera-angle-independent checks instead — shader compile/link success, geometry/shape correctness, uniform value plumbing, etc. — where a synthetic scene can't produce a misleading result.
**Source:** v1.7 floor-specular investigation, 2026-07-16; see `Scripts/dev_shader_tuning.py`

## `invoke()`/`Sequence` deferred callbacks freeze under `application.time_scale=0` (the pause mechanism) — 2026-07-09
**Context:** v1.7 pause-menu crash. Every menu button is a `ThemedButton` (`Scripts/ui_theme.py`) that plays a press animation then fires its real `on_click` via `invoke(self._fire_callback, delay=0.1)`. The pause menu pauses the game with `application.time_scale = 0` (main.py:957) — the project's pause mechanism.
**Symptom:** A pause-menu button click appears to do nothing while paused, then all frozen clicks fire in a burst the instant time resumes — the observed crash was Continue's `resume_game` immediately followed by Main Menu's `return_to_menu()` teardown running against now-stale/torn-down state, because both were queued during the same paused window.
**Root cause (confirmed by reading `ursina/sequence.py:120-133` + headless repro):** A plain `invoke(fn, delay=...)` builds a `Sequence` that advances on the **scaled** clock: `self.t += time.dt` (which is `time.dt_unscaled * application.time_scale`). With `time_scale=0`, `time.dt == 0`, so the sequence's `t` never reaches `delay` and the deferred `Func` never fires — *until* `time_scale` is restored, at which point every queued sequence elapses together. **Verified headless** (`window_type='none'`, `time_scale=0`): a plain deferred callback fired 0 times across 30 frames; the fixed version fired within one real frame.
**Fix (shipped):** `invoke(self._fire_callback, delay=..., ignore_paused=True)`. `ursina/ursinastuff.py:invoke` forces `unscaled=True` whenever `ignore_paused=True`, so the sequence advances on `time.dt_unscaled` (real time) *and* skips the `application.paused` early-return in `Sequence.update` — it fires promptly in real time regardless of `time_scale`. This is correct specifically because pause-menu buttons exist to act *on* the paused state. Two companion guards close the crash window: a class-level `_transition_pending` lock so a second click in the same paused window is a no-op while the first is pending (prevents two conflicting transitions being queued at once), and holding the returned `Sequence` on the instance so `on_destroy()` can `kill()` it — otherwise a destroyed button's pending callback survives free-floating in `application.sequences` (it holds a bound-method ref to the dead button) and fires later against stale state.
**Gotcha for the whole codebase:** any `invoke()`/`animate()`/`Sequence` that must run *while the game is paused* (menus, overlays, anything layered over `time_scale=0`) needs `ignore_paused=True`. Scaled-clock timers silently freeze under pause — they don't error, they just never fire, then burst on resume. **General repro discipline:** a tight `app.step()` loop advances `dt_unscaled` by ~0 real seconds between steps, so real-time (`ignore_paused`/`unscaled`) timers won't elapse in a headless harness unless you insert real wall-clock between frames — added `GameTestHarness.real_step()` (busy-waits ~20ms/frame) and the `pause_stacked_clicks` scenario for exactly this. The four pre-existing smoke scenarios all step against scaled game time, so none of them could ever have caught this — a click made under `time_scale=0` was an untested window.
**Source:** v1.7 pause-menu crash fix; `Scripts/ui_theme.py` `ThemedButton`; `tests/smoke_test_harness.py` `scenario_pause_stacked_clicks`. See also the animate() self-cancel gotcha below (same `invoke`/`Sequence` timing family).

## Two same-frame `animate()` calls on one property self-cancel via the shared animator + `interrupt='kill'` — 2026-07-09
**Context:** v1.7 weapon recoil. `Weapon._play_shoot_animation` (`Scripts/weapon.py`) did the classic "punch out then ease back" with two calls per property: `animate_z(rest-0.2, duration=0.05)` then `animate_z(rest, delay=0.05, duration=0.15)` (and the same pair for `rotation_x` kick).
**Symptom:** No error, but the punch/kick phase never visibly ran — the viewmodel animated rest-to-rest. Measured headless: z stayed at rest (z_min == rest, never reached rest−0.2), `rotation_x` never reached the −8 kick.
**Root cause (confirmed by reading `ursina/entity.py:animate` + headless repro):** `animate()` stores each property's tween in one per-property slot, `<name>_animator` (e.g. `z_animator`), and defaults to `interrupt='kill'`. The interrupt runs at the top of `animate()` — `getattr(self, animator_name).kill()` — **at creation time**, not when the tween starts interpolating. So the *second* (return) call, created in the same frame as the first (punch), kills the punch tween the instant it's constructed. The return call's `delay=0.05` only postpones when *it* starts interpolating; it does nothing to defer the interrupt. Net: punch tween is created then immediately killed, so it never advances, and the property animates straight rest→rest.
**Fix (shipped):** pass `interrupt='finish'` to each *return* call. `animate()` then calls `finish()` (not `kill()`) on the punch animator at creation — `Sequence.finish()` sets `t = duration` and calls `update()`, snapping the property to the punch/kick **end value** (rest−0.2 / rest−8) synchronously before the return tween takes over and eases it back. **Verified** by binding the real `Weapon._play_shoot_animation` to a carrier Entity and measuring: z reaches +0.80 (exactly rest−0.2) and `rotation_x` reaches −8.00, then both ease to rest; the pre-fix version left z at rest and `rotation_x` at 0. The snap-to-peak (vs interpolating rest→peak over the ~0.05s punch) is the correct recoil feel — the punch is meant to be near-instant; the eased 0.15s settle is what reads on screen.
**Rejected alternative — deferring the return call's creation via `invoke(delay=punch_duration)`:** measured *worse* (punch only reached ~−0.02 of its −0.2 target) because the deferred creation is fragile against frame timing and the return starts interpolating from wherever the punch happened to be at that instant. It also drags in the `invoke()`/`Sequence` pause-timing footgun (see the entry above) for no benefit. `interrupt='finish'` is both more surgical (one kwarg per return call) and provably correct.
**General rule:** to chain two tweens on the *same* property in the same frame ("go there, then come back"), the second call must NOT default-`kill` the first. Use `interrupt='finish'` on the second call (snaps to the first's end value, then animates on), or genuinely defer the second call's *creation* until the first has finished — but never rely on `delay=` alone to sequence them, because `delay=` defers interpolation start, not the creation-time interrupt.
**Source:** v1.7 weapon recoil self-cancel fix; `Scripts/weapon.py` `_play_shoot_animation`. Same `interrupt`/`Sequence` family as the pause-menu `invoke()` gotcha above.

## panda3d-gltf imports .glb models with `TransparencyAttrib:dual` — opaque meshes bleed through walls — 2026-07-07
**Context:** v1.7 through-wall enemy report. `enemy-flying.glb` (first live .glb in the project, swapped in at `f4cbe21`) visibly poked through walls it stood near — the dome/antenna of the drone rendered *over* a wall's face at certain angles.
**Symptom:** The enemy mesh shows through opaque world geometry (walls/blocks) it's behind or flush against, most visible near the top of the mesh at grazing camera angles. Occlusion is *partly* correct (dead-on it hides fine) but breaks by angle/distance — the tell-tale of a transparency-bin ordering issue, not a full depth-off.
**NOT the obvious causes (both ruled out by measurement, not assumption):** (1) *Not a scale/oversize issue* — the visual mesh's world AABB is only 1.08u wide vs the 3u box collider (mesh is *smaller* than the collider), so it can't physically reach a wall the collider stops 1.5u short of; `calcTightBounds` on the loaded model confirmed this. (2) *Not a depth-test-off shader* — `getDepthTest()`/`getDepthWrite()` on the visual model's Entity node both report `True`, same as a level_block.
**Root cause (found by walking the whole loaded subtree's RenderState, not just the top node):** panda3d-gltf sets `TransparencyAttrib:dual` (M_dual) on the model's **`ModelRoot`** node — a *child* of the Ursina `Entity`, so a `getTransparency()` probe on the Entity itself returns 0 and misses it. `M_dual` routes the mesh into the **transparent render bin**, drawn after all opaque geometry and outside normal opaque depth-sorting against walls, so it composites on top of walls it should be behind. The attrib comes from the glTF material's alpha handling (alphaMode / base-color alpha channel), applied at import regardless of whether the mesh is actually translucent.
**Confirmed via A/B screenshot** (headless smoke harness + `base.win.saveScreenshot`, same camera+wall): `M_dual` → drone bleeds through the wall; `M_none` → clean occlusion. Decisive, single-variable.
**Fix (shipped, `20e1b99`):** after constructing the visual child, force the whole subtree to `TransparencyAttrib.M_none` — `self.visual_model.setTransparency(M_none)` plus a loop setting it on each child (the attrib lives on the `ModelRoot` child, so the parent-only call is insufficient). This puts the opaque mesh back in the opaque pass to depth-test like every other world entity. Collider, facing offset (`rotation_y=180`), and mid-collider position untouched. See `Enemy.__init__` in `Scripts/enemy.py`.
**General rule for future .glb/.gltf assets:** any imported glTF mesh that is meant to be opaque needs its transparency forced to `M_none` after load, and you must **walk the loaded subtree** (not just probe the top Entity node) to see the importer's real render state — the interesting attribs sit on inner `ModelRoot`/`GeomNode` children. Same debugging discipline as the weapon depth-order bug: probe the actual live NodePath state, don't assume from the Python-side API. Related family: the viewmodel gun self-occlusion + black-gun-materials import issues.
**Source:** [[v1.7-editor-ux-scoping]]; `Scripts/enemy.py` `Enemy.__init__`; commit `20e1b99`

## Ursina/Panda3D `Vec3` has no `.rotate()` method — use `panda3d.core.Quat` — 2026-06-30
**Context:** v1.5 System B Step 9 (Shotgun pellet spread). Wanted to jitter `camera.forward` by a small random angle per pellet — the natural-looking API would be `direction.rotate(axis, degrees)`.
**Symptom:** Would have been `AttributeError: type object 'Vec3' has no attribute 'rotate'` at the first shotgun fire — caught before shipping by checking `dir(Vec3(...))` in a REPL first, not by writing the call and running it.
**Root cause:** Ursina's `Vec3` is a thin alias over Panda3D's `LVector3f`/`LVecBase3f`, which has no built-in axis-angle rotation method. `dir()` on a real instance confirms: no `rotate`, `rotate_around`, or similar — only component accessors, `cross`, `dot`, `normalized`, etc.
**Fix:** Build the rotation with `panda3d.core.Quat`: `q = Quat(); q.setFromAxisAngle(degrees, Vec3(axis)); rotated = Vec3(q.xform(direction))`. Verified working via REPL (`Quat().setFromAxisAngle(5, Vec3(0,1,0)).xform(Vec3(0,0,1))` returns the correctly-rotated vector). See `Shotgun._spread_direction()` in `Scripts/weapon.py`.
**Source:** [[v1.5-gameplay-systems]] (System B Step 9)

## `BUILTIN_MODELS[-1]` as an implicit "the special one" reference breaks the moment a new entry is appended — 2026-06-30
**Context:** `LevelEditor._make_trigger_entity()` read its tint colour from `self.BUILTIN_MODELS[-1]['color']` — correct only because Trigger happened to be the last entry in `BUILTIN_MODELS` at the time System A was written (v1.5 Step 6).
**Symptom:** Would have silently broken the moment System B Step 13 appended a `Pickup` entry after `Trigger` — `_make_trigger_entity()` would have picked up Pickup's cyan colour for trigger volumes instead of orange, with no error (list indexing never raises here — index -1 always resolves to *something*).
**Root cause:** Using negative list-index position as a stand-in for "the most-recently-added special entry" is an implicit ordering dependency that isn't enforced anywhere — nothing stops a later addition from being appended after it.
**Fix:** Give any `BUILTIN_MODELS` entry that other code depends on for non-card purposes (colour, scale, model) its own named class constant (`_TRIGGER_COLOR`), or look it up by `type` key via a derived `BUILTIN_MODELS_BY_TYPE = {a['type']: a for a in reversed(BUILTIN_MODELS)}` dict instead of position. Never reference `BUILTIN_MODELS[-1]`/`[0]`/any fixed index from code outside the asset-browser card-rendering loop (which legitimately iterates the whole list in order).
**Source:** [[v1.5-gameplay-systems]] (System B Step 13)

## `start_game()`'s enemy-placeholder loop reads `e.name` without the `_is_live()` guard — 2026-06-30 (fixed, latent/not human-reachable via shipped input flow)
**Context:** Discovered via a headless smoke test driving `main.py`'s flow programmatically (main_menu() → click Play → ...). Not introduced by any recent feature work — identical unguarded code existed on `version_1.5` HEAD already.
**Symptom:** `Assertion failed: !is_empty() at line 2102 of built1.10/include/nodePath.I` in `start_game()`, on `for placeholder in [e for e in scene.entities if e.name == 'level_enemy']:` — same C++ NodePath assertion class as Hard Constraint 13 / the original "NodePath teardown assertion (R-key crash)" gotcha below.
**Root cause (confirmed via isolated repro):** `main_menu()` calls `load_level()`, which destroys pre-existing `level_block`/`level_enemy`/`level_trigger` placeholders synchronously (their NodePaths empty immediately via `removeNode()`, but list removal from `scene.entities` is deferred to `scene._entities_marked_for_removal`, flushed only on the next frame's `taskMgr.step()` — see `ursina/main.py:192-194`). If `start_game()` runs before any frame flushes that queue, the enemy-loop's list comprehension evaluates `e.name` on *every* entity in `scene.entities`, including the just-emptied placeholders — tripping the assertion regardless of whether they'd match the `level_enemy` filter. **Confirmed exploitable with `main_menu()` immediately followed by `start_game()` in one synchronous call, zero frames rendered in between.** A parallel/separate mechanism exists on the *replay* path (`game.player` already set → `_clear_gameplay_entities`-style teardown at the top of `start_game()` destroys the old player/inventory/health_bar synchronously) but that path did NOT reproduce the crash when a real frame boundary (`app.step()`) separated the stages — only the zero-frame first-play case did.
**Human-reachability: NOT reachable via the shipped input flow.** Ursina's main loop flushes the deferred-removal queue every frame; a human always sees at least one rendered frame of the menu before they can physically click Play, so the click's `on_click()` dispatch always happens after at least one flush. The R-key return-to-menu path (`main.py` global `input()`) rebuilds the *menu* only — it doesn't call `start_game()` in the same handler, so there's always a frame between menu-rebuild and a subsequent Play click too. This is a **latent defect**: safe today only because of how the two calls happen to be wired to separate input events, not because of any guard. Any future change that chains `main_menu()` directly into `start_game()` in one call (auto-start, a "restart" shortcut, scripted/test driving) would hit it immediately.
**Fix:** Added the same `_is_live(e)` filter the codebase already uses everywhere else that reads `e.name` on `scene.entities`: `for placeholder in [e for e in scene.entities if _is_live(e) and e.name == 'level_enemy']:` (mirrors the `level_trigger` loop's existing guard four lines below it in the same function). Verified via two isolated repro scripts: (1) `main_menu()` → immediate `start_game()`, zero frames — crashed before the fix, clean after; (2) full first-play → win → return-to-menu → replay cycle with real frame boundaries — clean both before and after (confirms the replay path specifically wasn't the reproducible one, isolating root cause to the zero-frame first-play case).
**Source:** v1.5 HUD/menu redesign verification session; fixed same session. See also "NodePath teardown assertion (R-key crash)" below for the original instance of this bug class.

## `Entity.intersects(other)` — the arg is a traversal target, not the entity to test against — 2026-06-30
**Context:** v1.5 System A. The spec's `TriggerZone` (and `AmmoPickup`, Step 12 in System B) pseudocode reads `inside = self.intersects(player).hit` — meaning "am I overlapping the player?". This looks obviously correct and would be copied verbatim by anyone implementing from the spec.
**Symptom:** No crash, but wrong behaviour — the trigger fires (or never fires) regardless of where the player is. `self.intersects(player)` does NOT test "self vs player": Ursina's `Entity.intersects(traverse_target=scene, ...)` treats its first positional arg as the *root of the collision traversal*, then returns a `HitInfo` for whatever `self`'s collider hits within that subtree. Passing the player as `traverse_target` traverses only the player's node, so `.hit`/`.entities` reflect the player subtree, not a self-vs-player test — and `.hit` can read true/false for reasons unrelated to actual overlap.
**Root cause:** API shape mismatch. `intersects()` is "what does my collider hit when I traverse *this target*?", not "do I overlap *this specific entity*?". There is no built-in "do these two AABBs overlap" call with that signature.
**Fix:** Call `self.intersects()` with no arg (traverses the default scene), then test membership: `hit_info = self.intersects(); inside = hit_info.hit and player in hit_info.entities`. This is the live, correct pattern in `TriggerZone.update()` ([[v1.5-gameplay-systems]], commit `e93ebae`). **Treat this as the documented pattern for the whole codebase** — any future `.intersects(specific_entity)` call (notably the `AmmoPickup` in System B Step 12) needs the same correction. Do not copy the spec's pseudocode literally.
**Source:** [[v1.5-gameplay-systems]]; commits `e93ebae` (TriggerZone), `61b2712`/`cbd05d6` (v1.5 step-2 fixes)

## FPS gun viewmodel clipping — depth-state tricks fail, need a 2nd camera — 2026-06-30
**Context:** The FPS gun (`Weapon`, `parent=camera`, local `position=(0.5,-0.5,1)`) clipped through walls and level blocks when the player stood flush against them.
**Symptom:** The gun's far end pokes through / is painted over by world geometry as the camera approaches a wall. The geometry physically intersects the wall in world space — the gun is ~1.5u ahead of the camera.
**Root cause:** This is a depth-*order* problem caused by overlapping geometry, not a draw-order problem. Depth-state-only fixes — Ursina `always_on_top`, `setBin('fixed', 100)` + `setDepthTest(False)` (the gizmo pattern in Hard Constraint 11) — CANNOT fix it, because the gun shares the world's render pass: other world geometry in the same pass still paints over it even with depth-test off.
**Fix:** Standard Unity/Unreal dual-camera viewmodel. Give the gun a dedicated draw bit (`VIEWMODEL_MASK = BitMask32.bit(7)`), clear that bit from the main camera's mask so it renders everything except the gun, then add a second Panda3D camera (mask = bit 7 only, reuse `base.camLens`, parent to `base.cam`) on its own `make_display_region()` at sort 15 (after world=0/render2d=10, before UI=20). Hide the gun from all masks, show it only to `VIEWMODEL_MASK`. Because the VM pass runs after the whole world pass, the gun lands on top regardless of wall proximity. See `_setup_viewmodel_camera()` in `Scripts/weapon.py`.
**Footgun within the footgun (SUPERSEDED — see the 2026-07-07 correction below):** The original note here said to leave the display region's clear-depth OFF and run the gun with `setDepthTest(False)`/`setDepthWrite(False)`, claiming `set_clear_depth_active` "blanked the world" on macOS GL 2.1. That blanket depth-off is what later caused the gun's own sub-parts to render out of order (see the correction). The GLSL 1.20 shader patch in `main.py` rewrites shader *source* only, not render state, so it does not touch depth-test on the gun.
**Source:** [[v1.5-gameplay-systems]]

## Viewmodel gun sub-parts render out of depth order — the real fix is a depth-only region clear, NOT depth-off — 2026-07-07
**Supersedes** the "Footgun within the footgun" line in the entry directly above.
**Context:** After the v1.5 dual-camera viewmodel shipped with the gun set to `setDepthTest(False)`/`setDepthWrite(False)` and the VM display region NOT clearing depth, the real gun models (`Pistol_1`/`Shotgun_1`/`AssaultRifle_1`, wired in v1.7) rendered with their own sub-parts out of order — rear/interior faces painting over the front of the slide/receiver. Reproduced via the headless screenshot harness (drive `main.py` through `tests/smoke_test_harness.py`, spawn a `Weapon`, `base.win.saveScreenshot(...)`); most visible when the gun is turned broadside — a bright sliver near the barrel and light-gray receiver faces bleed through the near silhouette.
**Root cause:** With depth-test OFF, the gun's geometry paints in mesh-submission order (painter's algorithm). Each gun `.obj` is a **single merged Geom** (all 4–5 `usemtl` material groups collapse into one `mesh_geom` GeomNode — verified by walking the NodePath: one geom child, no separable per-material nodes), so its overlapping triangles have no way to sort against each other without the depth buffer. This is a *self*-occlusion bug within the model, distinct from the model-vs-world clipping the dual camera already solved.
**Why the old "clear-depth blanks the world" finding was wrong:** All display regions share ONE framebuffer; the *window* (`base.win`) clears colour+depth once per frame, then each region draws in sort order. Probing the live app (`win.get_display_region(i).get_clear_*_active()`): world region sort 0 clears nothing, render2d sort 10 clears depth, VM sort 15 cleared nothing, UI sort 20 clears nothing. Enabling **depth-only** clear on the VM region (`set_clear_depth_active(True)`, colour-clear stays OFF) wipes only depth for that pass — the world's already-drawn *colour* is untouched. Verified by screenshot: world fully intact, gun self-sorts correctly. The original blank was almost certainly from clearing *colour* (or the whole window), misattributed to depth.
**Fix (shipped):** In `_setup_viewmodel_camera()` — `dr.set_clear_depth_active(True); dr.set_clear_depth(1.0)` on the VM region (colour-clear left OFF). In `Weapon.__init__` — `setDepthTest(True)`/`setDepthWrite(True)` on the gun. The region gets a fresh depth buffer, the gun's parts sort against each other, and the whole gun still composites on top of the world (world depth wiped for this pass, world colour preserved). Verified across Pistol/Shotgun/Rifle at multiple yaw angles + an isolated large sky-facing shot: correct self-occlusion, world not blanked.
**Why NOT the `setBin` fallback:** per-part `setBin('fixed', N)` (the gizmo/crosshair pattern in Hard Constraint 8) is **impossible** here — the gun is one Geom with no child nodes to bin. Splitting the OBJ by material would be the only way, and the depth-buffer fix makes that unnecessary.
**Crosshair (shares nothing but the framebuffer):** The `PlayerHUD.crosshair` (`main.py`, `camera.ui`, sort-20 UI region, `setDepthTest(False)`+`setBin('fixed',100)`) is UNAFFECTED — it ignores the depth buffer entirely, and it draws in a later region than the VM pass. Verified two ways: (1) center-region pixels byte-identical before vs after the change; (2) toggling the VM region's `set_clear_depth_active` on/off in one live process → zero pixel difference at the crosshair. The main.py comment about the VM region "leaving the depth buffer populated with no clear" is now stale in wording (the region DOES clear depth now) but the crosshair's own depth-off still makes it robust — left as-is since behaviour is provably unchanged.
**This display region has now caused three issues** (crosshair depth-test loss, the viewmodel comment correction, and this sub-part ordering) — its real constraint is simply: *depth-only clear is safe, colour clear is not; the gun needs depth-test ON.*
**Source:** [[v1.7-fix-backlog]]; `Scripts/weapon.py` `_setup_viewmodel_camera()` + `Weapon.__init__`

## Ursina destroy() does not call entity.destroy() — override is dead code — 2026-05-20
**Context:** Tried to override `HealthBar.destroy()` to auto-clean up sub-entities parented outside the hierarchy (text parented to camera.ui instead of the health bar).
**Symptom:** Sub-entities survive after the parent entity is destroyed; text elements leak on scene transition.
**Root cause:** Ursina's `destroy()` function operates at the Panda3D NodePath level — it removes the entity from its parent's children list and schedules NodePath removal. It does NOT call any Python method on the entity object. A `def destroy(self)` override on an Entity subclass is unreachable via `destroy(entity)`.
**Fix:** Use `AliveEntity.on_die()` for cleanup in managed entities. Never rely on `entity.destroy()` being invoked. Explicitly call `destroy(sub_entity)` at every call site that needs cleanup.
**Source:** [[work/audits/2026-05-20-full-audit]]

## `camera.world_to_screen()` can raise or return None on off-screen entities — 2026-05-20

## InputField focus swallows keyboard input — 2026-05-20

## `Text(text='')` triggers IndexError in Ursina's align() — 2026-06-24
**Context:** `_build_asset_browser()`'s scroll-arrow `Text` entities (and two latent sites in `_update_inspector`/`_show_status_notice`) constructed or mutated a `Text` entity's `.text` to an empty string to hide it.

## `load_model()` caches shallow copies — per-instance vertex data leaks across every entity sharing that model — 2026-07-16
**Context:** Probing Candidate L2's AO bake (ambient-occlusion values written directly into per-vertex color on the mesh). Candidate L2 was ultimately shelved for unrelated structural reasons, but writing per-vertex data exposed this caching behavior.
**Symptom:** Writing vertex colors (or any other per-instance vertex data) onto one `Entity(model='cube')` silently shows up on every other entity using the same model string — no error, just shared/bled state across instances that look independent in code.
**Root cause:** Ursina's `load_model()` (`mesh_importer.py:38`) caches loaded models by name and returns the cached `GeomVertexData` by default — a shallow copy, not a deep one. Multiple `Entity(model='cube')` instances end up backed by the same underlying vertex buffer. Mutating vertex data on one instance mutates it for all.
**Fix:** Pass `use_deepcopy=True` to `load_model()` (or whatever entity-construction path routes through it) whenever code needs to write per-instance vertex data. Without it, per-instance vertex writes (AO bakes, per-vertex tinting, procedural vertex animation, etc.) are unsafe on any model loaded more than once.
**General rule:** any future per-instance vertex-data work (baked AO, vertex paint, procedural deformation) must check whether the model is shared and force a deep copy — this will resurface any time someone reaches for `model.geomNode` / `GeomVertexData` writes on a stock-cached model.
**Source:** v1.7 Candidate L2 AO-bake probe, 2026-07-16 (candidate shelved for structural reasons; caching finding stands independently)

## `destroy()` on any Ursina light leaks it forever — the LightAttrib keeps it lit — 2026-07-17
**Context:** v1.7 shadow work. [[work/active/v1.7-lighting-scoping]] Candidate L3's probe (`tools/shadow_fbo_probe.py`) found `on_lights` climbing 1→2→3→4 across menu cycles and flagged the teardown contract as unresolved. It is now solved.
**Symptom:** Every `main_menu()` rebuild leaves one more lit orphan (`-PandaNode/directional_light` — the leading `-` means its parent is detached) plus, for shadow casters, one more 1024x1024 depth FBO held forever. Silent: no error, nothing in `scene.entities`, and the scene looks correct because the extra suns are co-located. Reproduced in the shipped `main_menu()` as `[1,2,3,4,5,6]` over six cycles.
**Root cause:** `DirectionalLight.__init__` does `node_path = self.attachNewNode(self._light)` then `render.setLight(node_path)`. `node_path` is a **local** — Ursina never stores it, so nothing ever calls the matching `clear_light()`. `destroy()` empties the *Entity's* NodePath, which detaches the light from the scene graph but leaves `render`'s LightAttrib holding its own reference: still lit, buffer still allocated.
**Two traps that make the naive fix fail:**
- `render.clear_light()` with **no argument** clears the entire LightAttrib (every light, not just yours) — it is not a targeted release. This is why the probe's `shadows=False + clear_light()` attempt still saw the count climb.
- You cannot pass a re-wrapped `self._light`: `render.get_attrib(LightAttrib).get_on_light(i).node() is light._light` is **False**. Panda hands back a fresh Python wrapper around the same C++ node, so identity-matching against `_light` silently matches nothing.
**Fix:** `Scripts/light_lifecycle.py:destroy_light()` — recover the NodePath from the *Entity's own children* (`light.find_all_matches('**/+Light')`), then `render.clear_light(np)` → `np.remove_node()` → `set_shadow_caster(False)` (the FBO is **not** released by `clear_light()`) → `destroy(entity)`. Order-independent: destroying the first of two live lights correctly leaves the second lit.
**Pattern must be `+Light`, never `+LightNode`:** Panda splits lights across two hierarchies — `AmbientLight <- LightNode <- Light` but `DirectionalLight/PointLight/Spotlight <- LightLensNode <- Light`. `+LightNode` matches ambient **only** and would silently leak the sun — i.e. exactly the light you care about.
**When touching this:** never `destroy()` a light; route it through `destroy_light()`. `main_menu()`'s sweep does this via `is_light(e)`. The editor's `light_sun` is a plain Entity proxy (no `_light`), so it correctly stays on the normal `destroy()` path. Regression test: `tests/test_light_lifecycle.py` (includes a control that disables the fix and asserts the leak reappears).
**Source:** v1.7 shadow lifecycle session, 2026-07-17; [[work/active/v1.7-lighting-scoping]]

## Antialiased glyph edges make a pixel-diff HUD test report false failures — 2026-07-17
**Context:** `tools/bloom_probe.py` gate 2 asserts the sort-20 HUD is untouched by the bloom post-process, by diffing HUD pixels with bloom on vs off. First run "failed" with 371 differing pixels and nearly produced the headline *"the scoping doc's UI-camera claim is wrong on real hardware"* — which was itself wrong.
**Symptom:** A byte-compare over glyph pixels reports a few hundred differences when a bloomed emitter sits behind the text, even though the UI region never enters the filter.
**Root cause:** The text shader antialiases each glyph edge by blending glyph colour against **whatever is behind it** (compat.py's `_TEXT_FRAG` samples a 4-tap box around each texel, spreading the edge over ~2px). An edge pixel is therefore *part scene* by construction. Bloom legitimately brightens the scene behind the glyph, so those blended pixels move — with the HUD itself never being filtered. The tell is the channel breakdown: on an azure glyph the RED channel moves `0.000 -> 0.004` while blue holds — white emitter light bleeding through a partly-transparent edge, not the azure glyph being post-processed.
**Fix:** Erode the glyph mask before comparing — keep only pixels ≥2px inside it. Measured decay, which is what makes 2 the right number rather than a guess: `r=1: 1817 interior, 6 changed, max 4/255` → `r=2: 681 interior, 0 changed, max 0.0/255` → `r=3: 117 interior, 0 changed`. Clean at r=2 and stays clean, i.e. the residue is exactly the AA band and nothing else.
**General rule:** any pixel-diff assertion over antialiased content (text, sprite edges, MSAA silhouettes) must erode the mask first, or it measures alpha blending rather than the thing under test. Also report the overlap count (how many test pixels actually sit over the effect) — otherwise a "pass" can be vacuous because the subject never overlapped the effect at all.
**Source:** v1.7 bloom Phase 1 (B2) verification, 2026-07-17; `tools/bloom_probe.py` gate 2

## A Geom with no `vertex` column draws ZERO pixels — silently, with the attribute bound — 2026-07-17
**Context:** v1.7 particle scoping. [[work/active/v1.7-particles-scoping]] Candidate P3's probe (`tools/particle_batch_probe.py`) — batched single-Geom GPU particles where a `#version 120` vertex shader computes `p = p0 + v*t + ½g*t²`. Cost most of the spike's debug time and nearly produced the headline *"custom vertex attributes don't work at GLSL 1.20"*, which is the opposite of the truth.
**Symptom:** A `GeomVertexArrayFormat` carrying only user-named columns (`spawn_pos`, `spawn_vel`, …) and NO `vertex` column renders nothing. Not dim, not clipped — zero lit pixels. There is **no GL error, no compile error, no link error**; the GSG log cheerfully reports `Active attribute spawn_pos ... is bound to location 0`. Every signal says success except the frame, which is empty.
**Why it looks like an attribute-binding failure:** unbound GLSL attributes read as 0, so "custom columns didn't reach the shader" predicts exactly the same empty/degenerate frame. The two hypotheses are indistinguishable from the screenshot — which is why the probe needs the isolation below rather than a verdict from the failing case alone.
**Root cause:** Panda3D's Geom munger needs a `vertex` column to issue the draw *at all*, even when the shader never reads `p3d_Vertex`. Position being computed entirely GPU-side does not exempt you.
**Fix:** always `arr.add_column(InternalName.make('vertex'), 3, Geom.NT_float32, Geom.C_point)`. Zeroed data is fine if the shader ignores it — it exists only to satisfy the munger.
**Isolation that settled it (the reusable move):** build the SAME Geom twice, once with the `vertex` column and once without, and diff the lit-pixel count. With: 11,025 px. Without: 0 px, `spawn_pos` still reported bound. That's a one-variable experiment separating "attribute didn't bind" from "draw never issued" — a distinction no amount of staring at the empty frame resolves.
**Source:** v1.7 particle P3 spike, 2026-07-17; `tools/particle_batch_probe.py:_make_particle_format()`

## `sync-video` makes every perf number in a harness a lie — and `clock-mode none` breaks the fix — 2026-07-17
**Context:** v1.7 particle P3 spike, step 2 (measuring the ambient-dust ceiling). [[work/active/v1.7-particles-scoping]]
**Symptom:** `tools/particle_batch_probe.py` reported a median of **16.666 ms for every particle count from 0 to 8000** — including the empty scene. Read naively that says "8000 GPU particles are free", which is a conclusion the harness had not earned and would have shipped as a headline.
**Root cause:** that's the 60 Hz vsync deadline, not the GPU. `app.step()` sleeps to hit the frame boundary, so all the particle work hides inside the idle time and every count reports the identical cap. The number is a property of the display, not the code under test.
**Fix:** `loadPrcFileData('', 'sync-video 0')` before `Ursina()`. Real spread appears immediately: 0.37 ms empty → 0.40 ms at 8000 quads, and the fill-rate test separates cleanly (0.34 → 1.22 ms for 100x the fill at identical count).
**The trap in the fix:** do NOT also set `clock-mode none`. It uncaps the frame rate too, but it **freezes `get_frame_time()`** — and any shader animating off `osg_FrameTime` then renders a static scene, so you'd be timing a frozen batch while believing you'd measured a moving one. `sync-video 0` alone uncaps *and* keeps the clock advancing (verified: `frame_time` 0.6238 → 0.6392 → 0.6460 across steps).
**General rule:** any harness reporting a suspiciously round, suspiciously constant frame time is measuring vsync. Sanity-check by timing an EMPTY scene — if empty costs the same as loaded, the number is the display's, not yours. Applies to `tools/shader_harness.py`, `tools/bloom_probe.py`, and anything future that times frames.
**Source:** v1.7 particle P3 spike, 2026-07-17; `tools/particle_batch_probe.py`

## `window.size`'s setter only *requests* a Cocoa resize — `update_aspect_ratio()` alone can rescale `camera.ui_lens` against a stale window size — 2026-07-17
**Context:** v1.7 shadow/bloom UI verification. `camera.ui_lens` (an `OrthographicLens`) stays at its default `(1.0, 1.0)` film size until Panda's `aspectRatioChanged` event fires and rescales it to match the real window aspect ratio.
**Symptom:** Frame-1 UI (buttons, HUD elements built immediately after `window.size = ...` at app/editor startup) could be laid out against the wrong aspect ratio — specifically on macOS/Cocoa, where `window.size`'s setter only *requests* an async resize rather than applying it synchronously.
**Root cause:** On macOS/Cocoa, setting `window.size` queues a `WindowProperties` change that Cocoa applies asynchronously. `base.win.getSize()` keeps reporting the pre-resize size until Panda actually processes that pending change — normally on the next `aspectRatioChanged` event. Calling `window.update_aspect_ratio()` immediately after setting `window.size` (before that event has fired) reads the still-stale `getSize()` and rescales `camera.ui_lens` against the wrong dimensions.
**Fix:** Call `base.graphicsEngine.renderFrame()` immediately before `window.update_aspect_ratio()` — this forces Panda to process the pending Cocoa resize synchronously first, so `getSize()` reflects reality before the lens is rescaled. Needed at both entry points independently: `main.py` and `Scripts/level_editor.py` each set `window.size` and build frame-1 UI in their own separate init path — same shape as [[gotcha_openal_audio_crash_macos|the OpenAL audio-crash fix]] initially missing `level_editor.py`'s separate init path.
**General rule:** any window-size-dependent setup (aspect ratio, UI lens, layout) that runs immediately after `window.size = ...` on macOS needs a `renderFrame()` pump first if it can't wait for the async `aspectRatioChanged` event. Check both `main.py` and `Scripts/level_editor.py` — they do not share an init path.
**Source:** v1.7 shadow/bloom verification session, 2026-07-17; `main.py`, `Scripts/level_editor.py`
