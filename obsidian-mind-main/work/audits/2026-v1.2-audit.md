# v1.2 Bug Audit — 2026-05-20

## Executive Summary

**19 findings total: 0 CRITICAL, 4 HIGH, 9 MEDIUM, 6 LOW.**

The two most impactful issues are:

1. **`game.state` stuck at `PLAYING` after editor `_exit_play_mode()`** (HIGH — Track 1/8): `_exit_play_mode()` bypasses `game.return_to_menu()` and calls `_clear_gameplay_entities()` directly, so `game.state` remains `Game.PLAYING` after exiting editor play mode. Any feature that guards on `game.state` will misbehave (Esc triggers PauseMenu in the editor, `update()` collision checks still run, etc.).

2. **`PlaceEntityCommand.redo()` is a silent no-op after undo** (HIGH — Track 3): `PlaceEntityCommand.execute()` is intentionally a `pass` (entity already placed). When `undo()` destroys the entity and removes it from `self.editor.blocks`, a subsequent `redo()` calls `execute()` again — which still does nothing. The destroyed entity is not recreated, so redo silently fails. The undo stack then holds a command referencing a destroyed NodePath, corrupting future undo/redo.

3. **201 `eternal=True` debug entities created per `Player` instance** (HIGH — Track 12): `draw_raycasts=True` by default. The `"diagonal" in side_name` branch in `generate_raycast_points()` never fires (diagonal sides are named `"front_right"` etc., not `"diagonal_..."`), so all 8 sides fall through to the `5×5` column loop. 8 sides × 5 rows × 5 columns + 1 ceiling = 201 `eternal=True` `Mesh` entities created at player spawn. These are disabled but persist in `scene.entities` through scene transitions until explicitly destroyed.

Clean tracks: Track 5 (Inspector Panel — minor), Track 9 (Prefs persistence — PASS), Track 11 (Rule compliance — clean with one noted issue).

## Track 1 — Game State Machine

### Finding 1.1 — Circular import via `game.return_to_menu()` is safe at runtime
Severity: LOW (note, not a bug)
File: Scripts/game.py
Line(s): 25
Description: `game.return_to_menu()` does `from main import _clear_gameplay_entities` lazily at call time. `main.py` imports `from Scripts.game import game, Game` at module level. Python's import system caches modules after first load — by the time `return_to_menu()` is called, both `main` and `Scripts.game` are fully initialized. No circular import at load time. Safe.
Recommended fix: None required.

### Finding 1.2 — `game.state` stuck at `Game.PLAYING` after `_exit_play_mode()`
Severity: HIGH
File: Scripts/level_editor.py
Line(s): 523–542
Description: `_exit_play_mode()` calls `_clear_gameplay_entities()` directly (or the fallback inline block), but never calls `game.return_to_menu()`. `game.return_to_menu()` is the only method that transitions `game.state` from `RETURNING_TO_MENU` → `MAIN_MENU`. After `_exit_play_mode()` completes, `game.state == Game.PLAYING` (set by `game.start()` in `_spawn_gameplay_from_snapshot()`). Consequences: (a) in the level editor context the Esc key handler in `main.py input()` checks `if game.state == Game.PLAYING` — this guard would fire if the editor was launched via `main.py` (which it isn't standalone), but the state is semantically incorrect; (b) `collision_manager.update()` in `main.py update()` runs fine regardless; (c) the state is wrong for any future code that checks `game.state` from the editor.
Recommended fix: After `_clear_gameplay_entities()` succeeds in `_exit_play_mode()`, add `game.state = game.MAIN_MENU` (or call a new `game.reset()` method). Alternatively, refactor `_exit_play_mode()` to call `game.return_to_menu()` instead of `_clear_gameplay_entities()` directly.

### Finding 1.3 — State can stay stuck at `RETURNING_TO_MENU` if `_clear_gameplay_entities()` raises
Severity: MEDIUM
File: Scripts/game.py
Line(s): 23–27
Description: `return_to_menu()` sets `state = RETURNING_TO_MENU`, calls `_clear_gameplay_entities()`, then sets `state = MAIN_MENU`. If `_clear_gameplay_entities()` raises an unhandled exception, `state` stays at `RETURNING_TO_MENU` permanently. Any subsequent `game.state == Game.PLAYING` or `Game.MAIN_MENU` check will fail. The `escape` handler in `main.py` checks `game.state == Game.PLAYING` — this will never fire again. The game is softlocked.
Recommended fix: Wrap in try/finally: `try: _clear_gameplay_entities() \nfinally: self.state = Game.MAIN_MENU`.

### Finding 1.4 — WIN state defined but never entered or handled
Severity: LOW
File: Scripts/game.py, main.py
Line(s): 5, 347–390
Description: `Game.WIN = 'win'` is defined but no code path ever sets `game.state = Game.WIN`, and no `input()` or `update()` handler checks for it. The state constant is dead code and may mislead future developers into believing win logic exists.
Recommended fix: Add a comment `# WIN: not yet implemented — player health→0 resets position instead` or track in CLAUDE.md Known Tech Debt.

### Finding 1.5 — `start_game()` guard for pre-existing player uses `hasattr` not `game.player`
Severity: MEDIUM
File: main.py
Line(s): 128–136
Description: `start_game()` guards old player cleanup with `if game.player:`, then `if hasattr(game.player, 'weapon'):`. This correctly uses `game.player`. However, the `destroy(game.player.weapon.crosshair)` on line 130 has no None guard — `Weapon.crosshair` is always created in `Weapon.__init__` so it cannot be None, but if a future refactor makes crosshair creation conditional, this will crash silently. Low risk today but worth noting.
Recommended fix: Add `if game.player.weapon.crosshair:` guard (pattern already present in `_clear_gameplay_entities()` line 62).

### Verdict: 3 issues (1 HIGH, 1 MEDIUM, 1 LOW) + 1 note

## Track 2 — `_clear_gameplay_entities()` Teardown

### Finding 2.1 — `_exit_play_mode()` fallback path leaks weapon, crosshair, and health_bar
Severity: HIGH
File: Scripts/level_editor.py
Line(s): 529–537
Description: The `except Exception` fallback in `_exit_play_mode()` manually destroys enemies and the player via `destroy(game.player)` but does NOT destroy `game.player.weapon`, `game.player.weapon.crosshair`, or `game.player.health_bar` / `game.player.health_bar.text`. If `from main import _clear_gameplay_entities` raises (e.g., run standalone without `main.py` on the path), the weapon, crosshair, and screen health bar survive. On next F5 → play → F5 cycle, a new player creates new weapon+crosshair; the old ones are orphaned.
Recommended fix: Mirror `_clear_gameplay_entities()` teardown order in the fallback block, or always call `game.return_to_menu()`.

### Finding 2.2 — `AliveEntity.die()` before blanket loop: PASS
Severity: (none)
File: main.py
Line(s): 87–96
Description: `main_menu()` first iterates `scene.entities` calling `e.die()` on all live `AliveEntity` instances, then runs the blanket destroy loop. Correct. `_clear_gameplay_entities()` also calls `e.die()` on enemies before blanket loop. Both paths correct.

### Finding 2.3 — Pooled bullets are NOT in teardown list: PASS
Severity: (none)
File: main.py
Line(s): 72–78
Description: Parked bullets sit at `y=-10000`. The blanket loop destroys by name only (`level_block`, `level_enemy`, `ground`, `main_sky`, `camera_pivot`). Pool bullets have no `name` that matches, so they are not included. Pool bullets are `AliveEntity` instances — the `main_menu()` pre-loop will call `e.die()` on any that are `alive`, returning them to the pool (position parked). Bullets with `_alive=False` (parked) are skipped by `if e.alive`. This is correct — do not add pool bullets to the blanket loop.

### Finding 2.4 — `HealthBar.eternal=True` persists through scene transitions (known tech debt not fixed)
Severity: MEDIUM
File: Scripts/health_bar.py
Line(s): 16, 25, 36, 48, 65
Description: `HealthBar` and its `bg`, `bar`, and `text` sub-entities all use `eternal=True`. When `main_menu()` calls `scene.clear()` (if that path were taken), these would survive. Currently `main_menu()` uses explicit name-based destroy, not `scene.clear()`, so teardown works as long as callers correctly call `destroy(health_bar.text)` then `destroy(health_bar)`. However, `eternal=True` on the 3D health bar's `text`, `bg`, and `bar` (parented to the enemy, not `camera.ui`) means if Ursina's scene teardown ever calls `scene.clear()`, they will leak. This is existing tracked tech debt.
Recommended fix: Remove `eternal=True` from `HealthBar` sub-entities — it is no longer needed now that explicit teardown is in place.

## Track 3 — Undo/Redo Stack

### Finding 3.1 — `PlaceEntityCommand.redo()` is a permanent no-op after `undo()`
Severity: HIGH
File: Scripts/undo_redo.py
Line(s): 14–15, 17–24
Description: `PlaceEntityCommand.execute()` is `pass` (entity is already created at push time). `PlaceEntityCommand.undo()` destroys the entity and removes it from `editor.blocks` / `editor.enemies`. When `redo()` is called, it calls `execute()` again — which is still `pass`. The entity is not recreated and is not re-added to `editor.blocks`/`editor.enemies`. `_refresh_hierarchy()` is not called. From the user's perspective: place a block → Ctrl+Z undoes it (correct) → Ctrl+Y does nothing (entity gone, no visual change). The undo stack now holds a `PlaceEntityCommand` with a stale `NodePath` reference. A further Ctrl+Z will attempt `destroy(self.entity)` on an already-destroyed entity — Panda3D raises `AssertionError` or silently skips on dead NodePath, depending on version.
Recommended fix: `PlaceEntityCommand.redo()` must recreate the entity from a stored snapshot (same data as `DeleteEntityCommand.undo()`) and re-add it to `editor.blocks`/`editor.enemies`. Alternatively, snap the entity data on `__init__` and in `redo()` create a new entity from it.

### Finding 3.2 — `DeleteEntityCommand.undo()` does not re-add missing `collider`
Severity: LOW
File: Scripts/undo_redo.py
Line(s): 44–59
Description: `DeleteEntityCommand.undo()` recreates the entity with `collider='box'`, which is correct. `_entity_snapshot()` captures position, texture, color, rotation, enemy_hp, enemy_type, is_enemy — all needed fields. The snapshot does NOT capture scale. Enemy placeholders have `scale=(1.5, 3, 1.5)` — after undo they will be recreated with default Ursina scale `(1,1,1)`. Block entities have `scale=(1,1,1)` (default) so blocks are fine. For enemies this is a visual regression after delete+undo.
Recommended fix: Add `'scale': e.scale` to `_entity_snapshot()` and pass `scale=self.entity_data['scale']` in `DeleteEntityCommand.undo()`.

### Finding 3.3 — Undo stack cleared on load/save: PASS
Severity: (none)
File: Scripts/level_editor.py
Line(s): 757, 800
Description: `save_level()` calls `self._history.clear()` (line 757). `load_existing_level()` calls `self._history.clear()` (line 800). Both ground-truth reset points clear the stack. Correct.

### Finding 3.4 — Stack NOT cleared on play-in-editor entry/exit: MEDIUM
Severity: MEDIUM
File: Scripts/level_editor.py
Line(s): 514–519, 523–542
Description: `_enter_play_mode()` and `_exit_play_mode()` do not call `self._history.clear()`. After play mode exits, the undo stack may contain `MoveEntityCommand`, `ChangePropertyCommand`, etc. from the editor session. This is intentional per spec (play-in-editor should not reset undo history). However, if any gameplay action during play mode leaves stale entity references in the stack (e.g., if an enemy die()s, and the stack holds a `ChangePropertyCommand` for that enemy), Ctrl+Z after exit will `setattr()` on a dead NodePath. In practice this only affects the health bar (which is on the player, not editor entities), so the risk is low in current code but will grow as more inspector fields are added.
Recommended fix: Document that stack is intentionally NOT cleared on play entry/exit. Add a `# DELIBERATE: play-in-editor preserves editor undo history` comment. For extra safety, consider filtering dead NodePaths before undo.

### Finding 3.5 — `deque(maxlen=50)` discard of oldest command: PASS
Severity: (none)
Description: When the deque is full and a new command is pushed, the oldest is silently discarded. This is correct behavior — the discarded command's entity reference is released. No memory leak or crash risk.

### Verdict: 1 HIGH, 1 MEDIUM, 1 LOW

### Finding 4.2 — Selection set not cleared before entity deletion in `DeleteEntityCommand.execute()`: LOW
Severity: LOW
File: Scripts/level_editor.py
Line(s): 650–659, Scripts/undo_redo.py 34–41
Description: When `Delete` is pressed, `self.selected.clear()` is called (line 657) AFTER `cmd.execute()`. `DeleteEntityCommand.execute()` calls `destroy(self.entity)`. Since `self.selected` still contains the entity during `execute()`, and `_deselect_all()` is not called, the entity's `_original_color` reference survives on the destroyed entity. This is harmless — `destroy(e)` defers removal until end-of-frame, and `selected.clear()` runs immediately after. No crash risk.
Recommended fix: Minor — could call `self._deselect_all()` before the delete loop to be semantically clean, but current code is functionally safe.

### Finding 4.3 — Box-select `camera.world_to_screen()` protected by try/except: PASS
Severity: (none)
File: Scripts/level_editor.py
Line(s): 204–213
Description: The try/except wraps the `world_to_screen()` call. `if screen_pos and ...` guards None return. Correct per the Gotcha documented in `brain/Gotchas.md`.

### Finding 4.4 — Right-mouse-drag box-select vs EditorCamera rotation conflict: MEDIUM
Severity: MEDIUM
File: Scripts/level_editor.py
Line(s): 720–726
Description: `right mouse down` starts box-select (`_box_selecting = True`). `EditorCamera` also uses right-mouse drag for rotation by default. Both handlers fire simultaneously on right mouse down. The box-select records `_box_start` and then on `right mouse up` calls `_finish_box_select()` — but the user is also rotating the camera. Result: every right-drag for camera rotation also selects entities in a box matching the drag. This was acknowledged in the Gotcha notes but is not visually signaled to the user (no distinct modifier key distinguishes rotate from box-select). In practice the selection is usually wrong and confusing.
Recommended fix: Require `Shift+right-drag` for box-select (to avoid conflicting with EditorCamera rotation), or disable EditorCamera rotation when box-select is active.

## Track 9 — Camera Bookmarks + Grid Snap Persistence

### Finding 9.1 — `_load_prefs()` uses stored `self._editor_camera` reference, not `scene.entities[0]`: PASS
Severity: (none)
File: Scripts/level_editor.py
Line(s): 446, 644–646
Description: Camera bookmarks are applied via `self._editor_camera.position/rotation = ...`, where `self._editor_camera` is set from the standalone block to the actual `EditorCamera` instance. Not using `scene.entities[0]`. Reliable.

### Finding 9.2 — Bookmark keys fire during `InputField` focus: MEDIUM
Severity: MEDIUM
File: Scripts/level_editor.py
Line(s): 625–647
Description: Bookmark save (`Ctrl+1` through `Ctrl+5`) and recall (`1` through `5`) are handled in `LevelEditor.input()` with no check for whether an `InputField` currently has focus. The Gotcha in `brain/Gotchas.md` documents that `InputField` focus swallows `Delete` and `Ctrl+Z` — but number keys `1`–`5` would be consumed by the `InputField` for text entry, not reaching `LevelEditor.input()`. However, the bookmark handlers at lines 625–647 check `if key == str(i)` which would only fire if `LevelEditor.input()` receives the event. If `InputField` focus suppresses the event, the bookmark silently does not save/recall — this is the expected behavior. The risk is the OPPOSITE: while the user types a position value in the field (e.g., "15"), the digit `1` might still trigger bookmark recall before the field captures it. This is Ursina-version-specific behavior.
Recommended fix: Add an `InputField`-focus guard: `if any(getattr(f, 'active', False) for f in self._insp_fields.values()): return` at the top of `LevelEditor.input()`.

### Finding 9.3 — `_save_prefs()` synchronous JSON write on `Ctrl+1`–`Ctrl+5`: LOW
Severity: LOW
File: Scripts/level_editor.py

## New Gotchas

**`PlaceEntityCommand.execute()` no-op breaks redo after undo**
Context: Command-pattern undo/redo for entity placement where entity is created before command is pushed.
Symptom: Ctrl+Z correctly removes a newly placed block. Ctrl+Y (redo) appears to do nothing — block stays gone, no error.
Root cause: `PlaceEntityCommand.execute()` is `pass` because the entity already exists at push time. On `undo()` the entity is destroyed. On `redo()`, `execute()` is called again — still `pass`. Entity is not recreated. The undo stack now holds a command with a stale NodePath reference.
Fix: For place commands, store a creation snapshot at push time. `execute()` must recreate the entity (on redo) and add it back to `editor.blocks`/`editor.enemies`. `undo()` should store the created entity ref for `execute()` to use on redo. See `DeleteEntityCommand.undo()` for the recreation pattern.
Source: [[work/audits/2026-v1.2-audit]]

**`game.state` not reset by `_clear_gameplay_entities()` — only by `game.return_to_menu()`**
Context: Level editor `_exit_play_mode()` calls `_clear_gameplay_entities()` directly to tear down gameplay entities.
Symptom: After exiting editor play mode, `game.state == Game.PLAYING`. Any code checking this flag behaves as if the game is still running.
Root cause: `_clear_gameplay_entities()` is a pure teardown function — it destroys entities but does NOT change `game.state`. Only `game.return_to_menu()` transitions the state to `MAIN_MENU`. Calling `_clear_gameplay_entities()` directly skips the state transition.
Fix: Always go through `game.return_to_menu()` for canonical teardown (it calls `_clear_gameplay_entities()` internally), OR explicitly set `game.state` after direct calls.
Source: [[work/audits/2026-v1.2-audit]]

**`"diagonal" in side_name` dead branch when sides are named `"front_right"` etc.**
Context: `generate_raycast_points()` in `player_controller.py` tries to create fewer raycast start positions for diagonal corners.
Symptom: 201 `eternal=True` entity visuals created per Player spawn instead of the intended ~121. All disabled (show_colliders=False) so no visual impact, but they exist in the scene graph.
Root cause: The diagonal side names are `"front_right"`, `"back_left"`, etc. — none contain the literal string `"diagonal"`. The `if "diagonal" in side_name` branch is always False. All 8 sides fall through to the 5×5 column grid branch.
Fix: Rename diagonal side names to include `"diagonal_"` prefix, or restructure the loop to avoid the string check entirely.
Source: [[work/audits/2026-v1.2-audit]]

**Block `rotation` field written to JSON but not applied on level load or play-in-editor**
Context: `_build_level_data()` correctly serializes `rotation` for blocks. `load_level()` in `main.py` and `_spawn_gameplay_from_snapshot()` in `level_editor.py` do not pass `rotation=` to the spawned `Entity`.
Symptom: Rotated blocks in the editor appear un-rotated in both the main game and play-in-editor mode. The JSON is written correctly but silently ignored on load.
Root cause: `rotation=tuple(entity_data.get('rotation', [0,0,0]))` was not added to the entity constructor calls in `load_level()` (main.py:35–44) or `_spawn_gameplay_from_snapshot()` (level_editor.py:549–558).
Fix: Add `rotation=tuple(entity_data.get('rotation', [0,0,0]))` to both entity constructors.
Source: [[work/audits/2026-v1.2-audit]]
