### 4c. Enemy spawn Y offset
[main.py:24](main.py:24) — `position=tuple(entity_data['position'])`, no offset. Editor's `load_existing_level` uses `origin_y=-0.5` ([level_editor.py:1127](Scripts/level_editor.py:1127)) on the placeholder so its visual base is at `entity_data['position'].y`. The actual `Enemy` does NOT set `origin_y` ([enemy.py:9–16](Scripts/enemy.py:9)), so its origin is the center of its `scale=(1.5,3,1.5)` cube — i.e. its visual center is at `position.y` and its base is at `position.y - 1.5`. **There IS a residual coordinate mismatch:** the editor placeholder sits with its base at JSON position; the live enemy floats with its center at JSON position. Visually, a live enemy will appear 1.5 units lower than the editor placeholder.

### Finding 4.1 — Editor placeholder uses `origin_y=-0.5` but live `Enemy` does not → in-game enemy 1.5 units lower than editor placement
Severity: MEDIUM
RICE: R=3 I=3 C=4 E=2 → Score=18.0
File: Scripts/enemy.py, Scripts/level_editor.py, main.py
Lines: enemy.py:9–16, level_editor.py:1127, main.py:23–29
Description: v1.2.1's fix removed the `+1.5` offset in `load_level()`. That fixed the upward drift. But the editor's placeholder uses `origin_y=-0.5` (so its visual base is at the stored Y), while `Enemy.__init__` does not set `origin_y` — its origin defaults to the cube center. With `scale=(1.5, 3, 1.5)`, the live enemy renders with its **center** at the stored Y, meaning the live enemy's feet are 1.5 units BELOW the editor's visual placement. Most level.json enemies are at `y >= 1.5` so this presents as enemies that look fine in the editor but spawn buried halfway into the ground in-game.
Recommended fix: Add `origin_y=-0.5` to `Enemy.__init__()`'s `super().__init__()` call ([enemy.py:9–16](Scripts/enemy.py:9)). One-line change. This makes the live and placeholder coordinate conventions identical.

### Finding 4.2 — `invoke(setattr, self, 'can_shoot', True, delay=...)` fires on dead enemy after teardown
Severity: MEDIUM (carry-over from v1.2 #11)
RICE: R=2 I=2 C=3 E=2 → Score=6.0
File: Scripts/enemy.py
Line: 80
Description: When an enemy fires, `invoke()` schedules `setattr(self, 'can_shoot', True)` after `shoot_cooldown` seconds. If the player presses F5 (or returns to menu) within 1.0 second of an enemy firing, the enemy is destroyed but the scheduled callback still fires. `setattr` on a destroyed NodePath: in Ursina 8.3.0 this silently sets a Python attribute on the wrapper (which still exists in memory) — no crash, but a dangling reference. Risk grows if any future pooling reuses enemies.
Recommended fix: Override `Enemy.on_die()` to cancel pending invokes, or wrap the callback: `invoke(lambda: setattr(self, 'can_shoot', True) if self.alive else None, delay=self.shoot_cooldown)`.

## Known Bug Fix Specs

### Fix Spec A — Hierarchy / inspector / tray scroll zooms EditorCamera (Finding 8.1)

**Root cause:** Ursina dispatches `input(key)` to every `Entity` that defines it. `return` in `LevelEditor.input()` does not consume the event for `EditorCamera.input()`. EditorCamera handles `'scroll up'`/`'scroll down'` to zoom (set `camera.world_z`).

**Strategy:** Suppress `EditorCamera` from acting on scroll events when the cursor is over any editor panel. Two viable approaches:

**Approach 1 — Disable EditorCamera entirely while over a panel** (simplest, recommended):

Add a helper:
```python
def _cursor_over_any_panel(self):
    if self._hier_panel and self._is_over_panel(self._hier_panel):
        return True
    if self._inspector and self._is_over_panel(self._inspector):
        return True
    if self._is_over_tray():
        return True
    return False
```

In `LevelEditor.update()`, dynamically gate the camera:
```python
def update(self):
    if not self._play_mode:
        if self._editor_camera:
            self._editor_camera.zoom_speed = 0 if self._cursor_over_any_panel() else 1.25
        # ... existing body ...
```
(`EditorCamera`'s default `zoom_speed` in Ursina 8.3.0 is `1.25`. Confirm at runtime; if different, capture original at startup: `self._original_zoom_speed = self._editor_camera.zoom_speed`.)

### Fix Spec B — Drag-and-drop ghost mis-tracks cursor (Finding 8.2)

**Root cause:** The ray-direction formula in `_update_ghost()` (lines 541–546) treats `mouse.x`/`mouse.y` as degree deflections multiplied by `0.01745` (deg→rad). They are not angles — they are normalized screen coordinates. Ursina 8.3.0 provides the correct cursor-ray surface differently from previous versions.

**Strategy:** Reuse the pattern already working in `update_model_preview()` ([level_editor.py:877–887](Scripts/level_editor.py:877)). It uses `mouse.hovered_entity` and `mouse.normal` — the same surface that places blocks on click. The drag ghost should follow the same logic.

**Concrete replacement for `_update_ghost()`:**
```python
def _update_ghost(self):
    asset = self._drag_asset
    if asset is None:
        return

    # Use the same cursor-ray surface that update_model_preview uses.
    # mouse.hovered_entity / mouse.world_point / mouse.normal are Ursina's
    # canonical mouse-pick API on 8.3.0; they DO work mid-drag.
    if not mouse.hovered_entity or not mouse.hovered_entity.collider:
        if self._drag_ghost:
            self._drag_ghost.visible = False
        return

    # Reject hovering the ghost itself
    if self._drag_ghost and mouse.hovered_entity is self._drag_ghost:
        if self._drag_ghost:
            self._drag_ghost.visible = False
        return

    pos = list(mouse.hovered_entity.position + mouse.normal)
    pos = self._snap(pos)
    if asset['type'] == 'enemy':
        pos[1] += 1   # spawn enemy floor-aligned given origin_y=-0.5

    asset_color = color.rgb(*asset['color'])
    ghost_color = Color(asset_color.r, asset_color.g, asset_color.b, 0.5)

    if self._drag_ghost is None:
        self._drag_ghost = Entity(
            model=asset['model'],
            color=ghost_color,
            scale=asset['scale'],
            position=pos,
            collider=None,                       # critical: ghost must not be hoverable
            name='editor_drag_ghost'
        )
        if asset['type'] == 'enemy':
            self._drag_ghost.origin_y = -0.5
    else:
        self._drag_ghost.visible = True
        self._drag_ghost.position = pos
```

**Why this is correct:**
- `mouse.hovered_entity` is updated every frame by Ursina's picker, regardless of mouse-button state.
- `mouse.world_point` / `mouse.normal` are populated for the hovered entity's surface.
- `collider=None` on the ghost ensures the picker skips it — without this, on subsequent frames the picker returns the ghost itself, the ghost's `position + normal` is computed relative to its own face, and the ghost walks away from the cursor.
- `update_model_preview()` already proves this API works in 8.3.0.

**Critical additional constraints to enforce in the fix:**
1. **Cleanup on F5 mid-drag** (Finding 9.7): `_enter_play_mode()` must call `self._cancel_drag()` before spawning gameplay.
2. **Cleanup on Ctrl+L** (Ctrl+L not bound today; if added in v1.2.4): `load_existing_level()` must call `self._cancel_drag()`.
3. **Suppress placement path** (already correct): `'left mouse down'` handler at line 969 short-circuits with `return` when a tile is under the cursor — the placement-on-empty-ground path at line 986 cannot fire on the same event.
4. **Suppress dragging from tray to tray**: `_commit_drag()` already calls `_cancel_drag()` when releasing over tray (line 979–980). Keep.

**Removable code:** lines 540–553 (the wrong `ray_dir` computation and manual `raycast(...)` call) can be deleted entirely.

## New Gotchas Discovered

### Ursina `input()` events broadcast to every Entity — `return` is not "consume"
Context: Tried to suppress mouse-wheel zoom in `EditorCamera` by `return`ing early from `LevelEditor.input('scroll up')`.
Symptom: List scrolls AND camera dollies simultaneously on every wheel tick over a UI panel.
Root cause: Ursina dispatches `input(key)` to every Entity that defines an `input` method. There is no per-event "handled" flag at the Python layer that suppresses subsequent dispatch. `return` only ends *this* handler.
Fix: Disable or gate the consuming entity (e.g., set `EditorCamera.zoom_speed = 0` while cursor is over a panel) instead of trying to consume the event.
Source: [[work/audits/2026-v1.2.3-full-audit]] Finding 8.1

### `mouse.x` / `mouse.y` are screen-normalized, NOT angular deflections
Context: Reconstructed a cursor pick-ray after `mouse.direction` was removed in Ursina 8.3.0 by treating `mouse.x`/`y` as degree angles times `π/180`.
Symptom: Drag-and-drop ghost lands at a world point near the cursor but consistently offset; user perceives the drag as broken.
Root cause: `mouse.x` ranges roughly `[-aspect/2, +aspect/2]` and `mouse.y` roughly `[-0.5, +0.5]` in Ursina's `camera.ui` normalized coords. Multiplying by `fov * π/180` produces wildly miscalibrated tilt angles.
Fix: Don't reconstruct the ray manually. Use `mouse.hovered_entity` + `mouse.world_point` + `mouse.normal` — these are populated by Ursina's built-in picker and work mid-drag.
Source: [[work/audits/2026-v1.2.3-full-audit]] Finding 8.2

### Ghost / preview entities must have `collider=None`
Context: Created a translucent ghost entity that follows the cursor for drag-and-drop placement. Gave it a collider so it would render correctly.
Symptom: After first frame, the picker returns the ghost itself; subsequent placement positions are computed relative to the ghost's own face and the ghost walks away from the cursor.
Root cause: Ursina's mouse picker returns the closest hit entity, including the ghost. With `collider='box'`, the ghost is hoverable.
Fix: `collider=None` on every preview/ghost entity. Already correct in current `_update_ghost()` (line 575) — keep that constraint when rewriting per Fix Spec B.
Source: [[work/audits/2026-v1.2.3-full-audit]] Fix Spec B

### `Enemy` without `origin_y` renders 1.5u below editor placeholder
Context: Editor's `load_existing_level()` sets `origin_y=-0.5` on placeholders so their visual base aligns with the JSON Y. `Enemy.__init__` does not set `origin_y`.
Symptom: Enemies appear visually correct in the editor; in-game they spawn with their feet buried in the floor, centered on the JSON Y.
Root cause: Mismatched `origin_y` between editor placeholder and runtime Enemy class. Editor uses `-0.5` (base-anchored); runtime uses default `0` (center-anchored). With scale Y=3, the runtime visual is 1.5 units lower.
Fix: Add `origin_y=-0.5` to `Enemy.__init__()`.
Source: [[work/audits/2026-v1.2.3-full-audit]] Finding 4.1
