# v1.6 — Level Editor Refactor

**Shipped 2026-07-07** (CHANGELOG [1.6.0]). The 4,169-line `Scripts/level_editor.py` monolith is now seven focused modules, zero behaviour change, verified step-by-step through the real-dispatch smoke harness.

## Goal (as planned 2026-06-26)

Split `level_editor.py` into smaller, navigable modules. Two motivations, both explicit:

1. **Human navigability** — a single 4k+ line file is hard to hold in your head or scan for a specific feature.
2. **Agent productivity** — smaller files per concern mean less context an agent needs to load to work on any one task.

Gated on v1.3–v1.5 shipping, the [[work/active/v1.6-fix-backlog|2026-07-06 audit]]'s closure pass, and Ivan's manual module-boundary review — all met before the split started.

## Answers to the Open Questions (from the planning note)

- **Where do module boundaries fall?** One collaborator class per self-contained UI/behaviour band (hierarchy, gizmo, browser, inspector, playmode), each holding an editor back-reference; the core keeps shared state (selection, lists, history, snap, `_play_mode`) and the input()/update() dispatchers with the v1.2.4 priority chain intact.
- **Does the code naturally separate?** Yes, with one contract to respect: `undo_redo.py`'s commands call `editor._refresh_*` delegators by name, so the core keeps one-line delegators with the original names. The `_play_mode` flag stays on the editor because core, gizmo, and browser all read it.
- **Mechanical or logic-touching?** Mechanical — method bodies moved verbatim (`self.X` → `ed.X` only for core-owned state). Three genuine fixes were folded in deliberately and logged (backlog items 4/10/15) plus one integration miss the in-app run caught (`_INSP_LABEL_WS` read off the editor after the constant moved).
- **Regression approach?** After every step: 113/113 unit tests, the 4 standing smoke scenarios (real Ursina dispatch), and step-specific in-app drivers — 163 checks total across steps 2–6 (24/24/40/51/24), including two full F5 round-trips with real state-restore assertions.
