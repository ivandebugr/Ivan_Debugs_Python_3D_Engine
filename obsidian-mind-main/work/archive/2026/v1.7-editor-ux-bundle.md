# v1.7 Bundle — Gizmo Precision + Panel Basics

Implements B1 + B2(field) + C1 + C2 from [[work/active/v1.7-editor-ux-scoping|the v1.7 scoping report]],
in Ivan's stated priority order (gizmo/selection feel first, then panel/layout), matching the
technical dependency that C2 needs C1's discrete panel-visibility states to actually save anything.
Shipped CHANGELOG [1.7.0]. 113/113 unit tests + all 4 `tests/smoke_test_harness.py` scenarios green
after every step; each step additionally verified via headless in-process simulation (a real
`LevelEditor` instance driven directly, no window/mouse) since this environment has no way to drive
the Panda3D GUI interactively.

## Steps shipped

1. **B1 — cursor-pinned plane-projection gizmo drag + hover highlight** (`c146be9`) — Replaced the
   velocity-based drag (`mouse_vel.dot(axis_screen) * 200.0`) with plane-projection: cast the cursor
   ray onto a camera-facing plane containing the drag axis at grab time, project each frame's hit onto
   the axis line, move by the delta from the grabbed offset. Fixes the actual "feels imprecise"
   complaint — verified headless that a single fast cursor jump and a slow multi-step drag to the same
   endpoint now produce identical world displacement (previously speed-dependent). Falls back to the
   old velocity model when the axis is near-parallel to the view direction. Added hover highlight
   (brighten tip to white on `mouse.hovered_entity` before a drag starts). New pattern logged:
   [[brain/Patterns#Plane-projection for cursor-pinned gizmo drag (ray-line closest point) — 2026-07-07]]

2. **B2 — rotation numeric Rot X/Y/Z field** (`114d815`) — Extended the inspector's Pos/Scale grid to
   a third Rot row, reusing `_apply_inspector_value`/`ChangePropertyCommand` unchanged. Confirmed via
   full git history search (including the pre-v1.6-split monolith) that no rotation UI ever existed —
   this closes a real gap present since v1.2, not a regression. Row pitch tightened 0.30→0.20 across
   three rows to fit above the Texture row without moving it.

3. **C1 — collapsible/hideable panels** (`6983d82`) — `Ctrl+H`/`Ctrl+I`/`Ctrl+B` or a `[H]`/`[I]`/`[B]`
   chevron button toggles hierarchy/inspector/browser. `_apply_layout` reclaims freed horizontal space
   via new `_effective_hier_w`/`_effective_insp_w` properties threaded through every layout call site
   that previously read the raw width constants (including browser's tab row / card grid / scroll
   indicators, so browser content re-centers when a side panel collapses). `_is_over_panel`/
   `_is_over_browser` now gate on `.enabled` so collapsed chrome stops intercepting viewport clicks.
   Play-mode restore now respects each panel's collapse state instead of force-reopening everything.
   **Scope note:** browser vertical space is hidden but NOT reclaimed on collapse — full height-reclaim
   cascades through card-grid/scroll-indicator row math extensively, closer to C3 territory; deferred.
   Hit one real footgun along the way: a bare `<`/`>`/`^`/`v` `Button` label crashes `Text.align()` even
   with `use_tags=False` (`Button` doesn't forward that kwarg to its internal `Text`) — logged as a new
   Gotcha, worked around with bracketed ASCII labels (`[H]` etc.) matching the hierarchy panel's
   existing `[+]`/`[-]` convention. New pattern logged:
   [[brain/Patterns#Effective-width property pair for collapsible camera.ui panels — 2026-07-07]]

4. **C2 — saved layout presets** (`e1c4ce6`) — `Ctrl+Alt+1-5` saves current panel visibility to a
   slot, `Alt+1-5` recalls it; mirrors the camera-bookmark (`_bookmarks`/`_save_prefs`) pattern
   exactly, persisted under a new `layout_presets` key in `editor_prefs.json`. Alt-modified because
   `Ctrl+1-5` and bare `1-5` were already fully claimed by camera bookmarks — verified no collision via
   direct `input()` dispatch with `held_keys` set. Panel widths are captured in the preset dict for
   forward-compatibility with C3 but aren't yet applied (currently fixed constants) — only visibility
   round-trips today.

## Deferred (parked, not silently dropped)

- **B2-ring** — a rotation *ring* gizmo (vs. the numeric field shipped here). ~1–1.5 days, medium risk
  (new geometry, new drag mode, thin-target picking against the v1.2.4 pick-priority chain).
- **C3** — resizable panels (drag a divider). ~1–1.5 days per edge; a prerequisite for doing browser
  vertical-space reclaim on collapse properly (C1's current gap).
- **C4/C5** — movable-dock panels and true Unity-style drag-to-dock. C4 is 2–3 days; C5 is a multi-week
  framework-scale project the scoping report explicitly flagged as likely not worth it for a solo game
  editor. Full cost/risk breakdown in [[work/active/v1.7-editor-ux-scoping]].
