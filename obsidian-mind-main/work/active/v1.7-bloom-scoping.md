## Part A — Current state (grounded in the code, measured today, not assumed)

- **Offscreen color buffers are PROVEN on this GL 2.1 context** — `tools/retina_harness.py`
  exercised a 2× color buffer end-to-end (2026-07-16). **Depth-format FBOs are proven too** —
  `tools/shadow_fbo_probe.py` (2026-07-17, the L3 gate). The driver risk for "render scene
  into a texture" is retired twice over; unlike L3, bloom has *no* unresolved driver question.
- **Ursina already ships the hook.** `camera.shader = <Shader>` (ursina `camera.py:145-180`)
  creates a `FilterManager(base.win, base.cam)`, calls `renderSceneInto(colortex, depthtex)`,
  and applies the shader to a fullscreen quad with `tex`/`dtex` inputs. One filter only, no
  chain — but the plumbing exists and Ursina owns its cleanup path.
- **The UI/HUD is architecturally separate** (the session's supersampling gotcha, verified in
  ursina `camera.py:43-81`): the 3D scene renders through `base.cam`'s display region
  (**sort 0**); all UI renders through `ui_camera` into its **own display region (sort 20)**
  on the same window. Any post-process that wraps only `base.cam` leaves the HUD region
  untouched.
- **How `FilterManager.renderSceneInto` composites (read from Panda 1.10.16 source):** it
  redirects `base.cam` into an offscreen buffer (buffer sort = `win.sort − 1000`, so buffers
  render *before* the window), installs the fullscreen quad **in the original sort-0
  region**, and disables that region's clears. The sort-20 UI region then draws directly to
  the window *after* the quad. **Net: the HUD composites on top of the post-processed frame,
  crisp and un-bloomed, by construction — the "naive wrap misses UI" failure mode is actually
  the desired outcome here, as long as nobody ever wraps `ui_camera` too.**
- **Resize is handled:** `FilterManager` self-subscribes to Panda's `window-event` and calls
  `resizeBuffers()` (FilterManager.py:78,344-348) — independent of the project's
  `window.update_aspect_ratio` wrapper (Hard Constraint #13). Needs a one-line verification
  in the spike, not new code.
- **☠️ Ruled out by measurement — Panda's built-in `CommonFilters.setBloom()`.** Its four
  bloom passes are all `Shader.make(..., Shader.SL_Cg)`. On this arm64 build,
  `Shader.make(<minimal Cg>, SL_Cg)` returns `None` with *"Support for Cg shaders is not
  enabled"* (measured 2026-07-17). The free built-in bloom is dead on Apple Silicon; every
  candidate below is hand-rolled GLSL 120. Logged to [[brain/Gotchas]].
- **Framebuffer reality check:** the window framebuffer is 1280×720 *physical* (retina
  harness finding — macOS upscales). Fill-rate math below uses 720p, and the OS upscale adds
  free extra softness to any blur.
- **LDR only, and that's fine:** no HDR float targets at 8-bit RGBA. TF2-era bloom (DX8/DX9)
  was LDR bloom with a luminance threshold — the look this project wants predates HDR
  pipelines anyway.

## Part B — What "bloom" decomposes into

Every real bloom is the same three steps; candidates differ in *where* they run and *what*
qualifies as "bright":

1. **Select** — which pixels glow (luminance threshold, or an authored glow mask)
2. **Spread** — blur the selection (the quality/cost fork: blur radius vs. resolution)
3. **Composite** — add the blurred glow back over the frame, under the HUD

## Candidate B1 — Single-quad "bloom-lite" via Ursina's own `camera.shader` hook

**What:** use the hook Ursina already ships. One `#version 120` fragment shader on the
filter quad: threshold + small fixed-tap blur (9–16 `texture2D` taps at full res) + additive
composite, all in a single pass. No new machinery — the engine-side work is one shader
string in the `Scripts/` style plus a `camera.shader = bloom_shader` call gated to gameplay.

- **Passes:** 1 scene buffer (color+depth — the hook allocates both) + 1 fullscreen quad.
- **GLSL 120:** trivial — `texture2D`, constant-bound loops, `mix`/`max`. Same vocabulary
  `compat.py` and `lit_shader.py` already use.
- **Perf (M3):** 720p × ~16 taps ≈ well under 0.5 ms. Ceiling unchanged (120 fps).
- **UI-camera answer:** wraps `base.cam` only; sort-20 UI region composites on top,
  un-bloomed (Part A mechanism). **Correct by default — nothing to do.**
- **Breaks / revisit:** the quality ceiling is the candidate's honest weakness — a full-res
  small kernel gives a tight neon edge-glow (a few pixels), *not* the wide soft TF2 halo;
  wide blur in one full-res pass needs unaffordable tap counts. `camera.shader` persists
  across scene sweeps (it lives on the camera, not the scene), so gate it on
  `game.state == Game.PLAYING` transitions and clear it for menu/editor/F5 — small, but it's
  this project's teardown lesson again.
- **Risk: LOW.** Effort ≈ 1 session. Cheapest path to *any* glow; wrong shape for painterly.

## Candidate B2 — Classic downsampled bloom chain (bright-pass → separable blur → composite)

**What:** the real thing — hand-rolled `CommonFilters` bloom shape in GLSL 120, owned by us.
`FilterManager.renderSceneInto` for the scene, then `renderQuadInto(div=4)` bright-pass,
blur-X, blur-Y at quarter res, and a composite quad. Blur radius comes from downsampling
(¼-res pixels are 4× wider), so a 9-tap separable Gaussian reads as a wide soft halo.

- **Passes:** scene buffer + 3 small quad passes (bright, blurX, blurY) + composite quad.
- **GLSL 120:** all core — threshold, constant-loop separable Gaussian, additive composite.
  Linear-filtered sampling of the smaller buffer does the downsample; no `texelFetch`, no
  extensions, no MRT.
- **Perf (M3):** bright+blur run at 320×180 (~57k fragments × ~9 taps each) — well under
  0.5 ms for the whole chain at 720p. Ceiling unchanged.
- **UI-camera answer:** identical mechanism to B1 — only `base.cam`'s region is redirected;
  the chain's intermediate buffers use their own quad cameras and never see `ui_render`.
  HUD stays crisp on top. **Correct by default.**
- **Breaks / revisit:** bypasses Ursina's single-filter `camera.shader` path — a manual
  `FilterManager` and the Ursina hook would fight over `base.cam`'s display region, so pick
  one owner (manual) and never mix. Owning the chain means owning buffer lifecycle: create
  **once at app init**, toggle by enabling/disabling the quads — never create/destroy through
  `return_to_menu()`, which is exactly where the shadow spike found the bodies buried
  ([[v1.7-lighting-scoping]] L3 findings). Editor viewport: decide bloom-on or bloom-off in
  editor and F5 play-mode explicitly.
- **Risk: LOW–MEDIUM** (lifecycle discipline, not driver risk — FBOs proven twice).
  Effort ≈ 1–2 sessions incl. a `shader_harness`-pattern spike. The "real bloom" candidate.

## Candidate B3 — Glow-map bloom (art-directed: only flagged things glow)

**What:** same spread/composite as B2, but **selection is authored, not thresholded** — TF2's
actual approach to glow discipline. Two wirings:

- **(a) Alpha-mask:** scene shaders write a glow amount into `gl_FragColor.a`; the
  bright-pass keys on alpha instead of luminance. Needs an alpha channel in the scene buffer
  and a `glow_amount` uniform (default 0) added to `compat.py`'s three shaders +
  `lit_shader.py`.
- **(b) Second glow camera:** a `base.cam`-slaved camera with a draw-mask renders *only*
  glow-flagged entities (`entity.hide(GLOW_MASK)` bookkeeping) into a ¼-res color buffer;
  blur that; composite. Zero scene-shader edits.

- **Passes:** (a) = B2's four. (b) = B2 + one extra scene render of a handful of flagged
  entities into a small buffer — still trivial.
- **GLSL 120:** fine in both; (b) barely needs new shader surface beyond the blur pair.
- **Perf (M3):** ≈ B2 either way; (b)'s glow pass draws few entities at ¼ res. Negligible.
- **UI-camera answer:** (a) inherits B2's correct-by-default story. (b) is even cleaner —
  the UI never enters *any* buffer — but the additive composite must land **above the scene,
  below the HUD**: give the composite quad its own display region at a sort between 0 and 20
  (e.g. 10). That's the one candidate where the UI answer is a deliberate wiring choice
  rather than a freebie.
- **Breaks / revisit:** (a) collides with transparency — alpha is a blending channel on
  transparent entities, so glow-in-alpha needs a convention (glow only on opaque geometry).
  It also widens `compat.py`'s patched-shader surface, the most delicate file in the repo.
  (b) trades that for draw-mask bookkeeping across pooled bullets (position-parked at
  y = −10000 — a flagged parked bullet still renders into the glow buffer; park location is
  off-frustum so likely moot, verify in spike) and editor proxies.
- **Risk: MEDIUM** (touches either every shader or entity bookkeeping). Effort ≈ 2–3
  sessions. Highest art-direction payoff: muzzle flashes, pickups, and the sun glow while
  white walls don't — which is what "TF2 bloom" actually means.

## Candidate B4 — No post-process: additive billboard glow sprites on emitters

**What:** the [[v1.7-lighting-scoping|L2]]-style illusion. No FBO, no passes — a soft radial
glow texture on additive-blend billboard quads attached to the things that should glow
(muzzle flash, pickups, sun corona). Pure content + entity work on the existing unlit shader.

- **Passes:** none. GLSL 120: n/a (existing shaders).
- **Perf (M3):** a handful of alpha-blended quads — zero.
- **UI-camera answer:** not applicable by construction — glow sprites are scene entities
  under `base.cam`, drawn like everything else; the HUD renders after them exactly as today.
- **Breaks / revisit:** no glow from bright *surfaces* (skybox edges, lit wall tops) — only
  from placed emitters. Billboards depth-test against geometry, so a half-occluded glow
  pops rather than fades; the standard fix (raycast visibility fade) is one small update
  hook. Pool-bullet muzzle flashes need the flash quad managed by the weapon, not the pooled
  bullet (pool bullets must never toggle `enabled` — Hard Constraint #6).
- **Risk: LOW.** Effort < 1 session. Composable with B1–B3 later; also the fallback if any
  post-process surprise appears.

## Summary matrix

| | Render passes | GLSL 120 | UI/HUD answer | Perf (M3, 720p) | Risk | Effort |
|---|---|---|---|---|---|---|
| **B1** single-quad lite | scene buffer + 1 quad | trivial | correct by default (sort-20 on top) | <0.5 ms | LOW | 1 session |
| **B2** downsampled chain | scene buffer + 4 quads | core only | correct by default (same mechanism) | <0.5 ms | LOW–MED | 1–2 sessions |
| **B3** glow-map | B2 (+1 small scene pass in (b)) | fine | (a) default · (b) **explicit sort-10 region** | ≈ B2 | MED | 2–3 sessions |
| **B4** billboard sprites | none | n/a | n/a by construction | ~0 | LOW | <1 session |

**Composition notes (not a recommendation):** B1 is a strict quality-subset of B2 sharing
one shader's worth of work — they're stages of one path more than rivals. B3 is B2's
selection stage swapped for authorship and is the only candidate whose *look* matches the
TF2 reference rather than "everything bright halos." B4 is orthogonal and composable, same
role L2 played in the lighting scoping. Unlike L3, **no candidate here carries driver risk**
— buffers are proven on this stack twice; the recurring cost across B1–B3 is the same
lifecycle discipline (create once at init, never through `return_to_menu()`) the shadow
spike just re-confirmed as this project's real failure surface. Interaction with
[[v1.7-lighting-scoping]]: bloom reads best *after* an L1-style grade exists (something has
to be bright before it can glow); if L3's shadow work ever lands, it shares nothing with
this machinery and neither blocks the other.

**Decision status:** none picked. If any of B1–B3 is in play, the first gate is a
`tools/`-harness spike (shader_harness pattern) proving the FilterManager quad + sort-20 HUD
compositing claim on the real windowed context — cheap, because unlike L3 the expected
answer is already "yes" twice over.
