## Part A — Current state (grounded in the code, not assumptions)

### The gizmo today (`editor_gizmo.py`)
- **Translate-only.** X/Y/Z line shafts + cube tips. `grep` for any rotation/scale
  gizmo returns nothing — it does not exist.
- **Velocity-based drag, not point-under-cursor.** `handle_drag()` computes
  `magnitude = mouse_vel.dot(axis_screen) * 200.0` and adds it to the axis each
  frame. Consequences:
  - The handle does **not** stay pinned to the cursor — it tracks *mouse speed
    along the screen-projected axis*, so a fast flick moves the entity further than
    a slow drag of the same screen distance. This is the single biggest "feels
    imprecise" source.
  - The `* 200.0` constant is a magic scale with no relation to world units or DPI.
  - Snap is applied per-frame (`_snap_1d(raw)`), so with snap on the entity jumps
    between grid cells as you drag — reasonable, but with snap **off** there's no
    fine control other than moving the mouse very slowly.
- **Pick is solid.** `cursor_ray()` correctly rebuilds the pick ray through the
  free cursor via `camera.lens.extrude` (not `camera.forward`); tips render on top
  via `setBin('fixed', 100)` + depth-test off. `try_begin_drag()` raycasts tips
  before any panel check (v1.2.4 priority chain). No bug here — this part is good.
- **No hover feedback.** Handles don't highlight on hover before you commit to a
  drag. The line meshes *do* report as `mouse.hovered_entity` (Gotcha 2026-05-20),
  so the hook to detect "cursor is near axis X" already exists and is unused for
  visual feedback.
- **Multi-select works, group-transform is naive.** Box-select (right-drag) and
  shift-click populate `self.selected`; `refresh()` centroids the gizmo correctly.
  But `handle_drag()` applies the *same per-axis delta* to every selected entity —
  fine for translate, but there is no group bounding-box, no pivot options, and
  (since there's no rotate/scale gizmo) no way to scale/rotate a selection about
  its centroid at all.

### Numeric input today (`editor_inspector.py`)
- Inspector grid is **Pos X/Y/Z + Scale X/Y/Z only**, Enter-to-apply
  (`_apply_inspector_value`, `submit_on=['enter']`). Each pushes a
  `ChangePropertyCommand` (undoable) and refreshes the gizmo. This works well.
- **Rotation has NO editor UI at all.** It round-trips through JSON
  (`_build_level_data` writes `rotation`, load reads it) and *was* silently dropped
  on the game-side load path — that specific bug is fixed (Gotcha 2026-05-20) — but
  there is no field or gizmo to *set* rotation in the editor. Blocks can only be
  rotated by hand-editing `level.json`. This is the biggest concrete gap.

### Panel layout today (`editor_core._apply_layout` + per-panel `apply_layout`)
- Three fixed-anchor panels, widths are hardcoded constants:
  - Hierarchy — flush **left**, `_LAYOUT_HIER_W = 0.20`
  - Inspector — flush **right**, `_LAYOUT_INSP_W = 0.30`
  - Asset browser — full-width **bottom** band, `_BROWSER_H = 0.20`
  - Toolbar — right-anchored strip, per-button widths scale with aspect ratio.
- On resize, `_apply_layout` re-derives every element's absolute position from the
  current `aspect`/`half_w`. This machinery is **solid and well-understood** — it's
  the payoff of the v1.2.6 resize work (Hard Constraints 12–13, wrapping
  `window.update_aspect_ratio` since `window.on_resize` is dead code).
- **No user control over layout:** panels can't be resized, collapsed, moved, or
  hidden individually. Widths are author-time constants.
- **Structural gotcha for any layout rework:** browser **cards are parented to
  `camera.ui` directly**, not to the browser panel (`_create_browser_card` →
  `parent=camera.ui`, positioned by `_card_grid_pos`). Same for scroll indicators.
  So "move the browser panel" today means "reposition the panel quad *and* every
  card by hand in `apply_layout`." Any docking/resizing scheme that assumes
  "reparent a panel and its children follow" does **not** hold here without first
  reparenting cards under their panel — a real, if bounded, cost.

## Part B — Selection & gizmo candidates (2–3, with cost/risk)

### B1. Point-under-cursor drag + hover highlight  ⭐ highest feel-per-effort
Replace the velocity model with **plane-projection drag**: on grab, cast the cursor
ray onto the plane containing the drag axis; each frame, project the cursor onto the
axis line and move the entity so the grabbed point stays under the cursor. Add a
hover-highlight (brighten the tip when `mouse.hovered_entity.name == editor_gizmo_*`
— the hook already fires).
- **Fixes:** the core "imprecise / speed-dependent" complaint; the handle finally
  sticks to the cursor like Unity/Blender.
- **Cost:** ~half a day. Contained entirely in `editor_gizmo.handle_drag()` +
  `cursor_ray()` (already exists). Math is a ray-line closest-point, no new deps.
- **Risk:** low-medium. Edge case: axis nearly parallel to view direction → tiny
  screen motion = huge world motion (standard gizmo problem; clamp or fall back to
  the current velocity model when `axis_screen.length()` is below a threshold — that
  guard already exists in the code).

### B2. Rotation gizmo + rotation numeric field
Add a rotation ring (or, cheaper, three axis-rotate handles) and a Rot X/Y/Z row in
the inspector grid mirroring the existing Pos/Scale rows.
- **Fixes:** the biggest *capability* gap — rotation is currently JSON-only.
- **Cost:**
  - Numeric field only (Rot X/Y/Z row): **~2–3 hours**, low risk. Reuses the exact
    `_apply_inspector_value` + `ChangePropertyCommand` path; just add `rot_x/y/z →
    rotation_x/y/z` to `attr_map` and three grid cells. **Recommended baseline** —
    high value, trivial risk.
  - Ring gizmo on top: **~1–1.5 days.** New ring geometry (Ursina has no torus
    primitive — build a line-loop `Mesh` per axis, or import one), plane-intersection
    drag math, screen-space angle delta, a new `RotateEntityCommand` for undo. Ring
    picking is fiddlier than cube-tip picking (thin target).
- **Risk:** field = low; ring = medium (new geometry, new drag mode, must not break
  the v1.2.4 pick priority chain).

### B3. Adjustable/finer snap + snap-to-surface
Snap increments today are `(1.0, 0.5, 0.25, None)`. Add finer steps (0.1, 0.05) and/or
a numeric snap entry; optionally a "snap dragged entity to the face under cursor"
mode (the `mouse.normal` used by place-mode already gives the face).
- **Fixes:** precision for players who like grid workflow; complements B1 for
  non-grid workflow.
- **Cost:** finer increments = **~1 hour** (extend `EDITOR_GRID_SNAPS`, relabel
  button). Numeric entry / surface-snap = ~half a day.
- **Risk:** very low.

> **Not recommended as first work:** group bounding-box scale/rotate about centroid.
> It only becomes meaningful *after* a rotate/scale gizmo exists (B2), and multi-entity
> pivot semantics are a rabbit hole. Defer until single-entity rotate ships and Ivan
> actually wants it.

## Part C — Panel/window layout: honest feasibility

**The headline:** Ursina's `camera.ui` is a fixed 2D overlay (a Panda3D `NodePath`
tree spanning y∈[-0.5, 0.5], x∈[-aspect/2, +aspect/2]). It is **not** a windowing
toolkit. There is no docking manager, no window server integration, no built-in
drag-to-dock, no floating OS windows. Everything below is "what we'd build on top of
a 2D overlay," and cost rises steeply with ambition. Laid out cheapest → most
ambitious so the tradeoff is explicit.

### C1. Collapsible / hideable panels — **cheap, high ROI**
Toggle each panel's `.enabled` (and reclaim the space in `_apply_layout` by widening
the viewport region). Hotkeys or small chevron buttons.
- **Cost:** ~half a day. Pure bookkeeping in the existing `_apply_layout`.
- **Does NOT give you:** resizing or repositioning — just show/hide.
- **Risk:** low. The one wrinkle is the browser's `camera.ui`-parented cards must be
  hidden with the panel (the `set_visible` cascade already handles this).

### C2. Saved layout presets (bookmark-style) — **cheap, fits existing pattern**
Store `{hier_w, insp_w, browser_h, which panels visible}` in `editor_prefs.json` and
recall via number keys — exactly like the existing camera bookmarks
(`_bookmarks` + `_save_prefs`). Best combined with C1/C3.
- **Cost:** ~2–4 hours once there's *something* to save (i.e. do after C1 or C3).
- **Does NOT give you:** free-form arrangement — presets only snapshot whatever
  discrete knobs exist.
- **Risk:** low.

### C3. Resizable panels (drag a divider) — **moderate**
A draggable splitter quad between panel and viewport that writes back into
`_LAYOUT_HIER_W` / `_LAYOUT_INSP_W` (promote them from class constants to instance
vars) and re-runs `_apply_layout`.
- **Cost:** ~1–1.5 days per edge. Splitter entity, drag handling in the input
  priority chain, clamps, persistence (folds into C2).
- **Does NOT give you:** reordering or undocking — panels stay in their fixed edge,
  they just get wider/narrower.
- **Risk:** medium. Must slot into the input priority chain without stealing gizmo
  or panel clicks. Browser cards need repositioning on width change (they already
  reposition in `apply_layout`, so this is continuous with existing code).

### C4. Movable-but-fixed-dock panels — **moderate-high**
Let a panel be reassigned to a different edge (left/right/bottom) from a menu, rather
than dragged freely. Discrete slots, not free-floating.
- **Cost:** ~2–3 days. Needs a slot model, an occupancy/conflict rule (two panels
  same edge → stack? reject?), and generalizing each panel's `apply_layout` to accept
  a target edge instead of hardcoded left/right.
- **Does NOT give you:** free-floating windows or drag-to-dock animation — you pick
  an edge from a list.
- **Risk:** medium-high. The browser is currently hard-wired to "full-width bottom";
  making it edge-agnostic is the most invasive piece.

### C5. True Unity-style docking (drag-to-dock, split regions, floating panels) — **expensive; feasible only in the weak sense**
- **Drag-to-dock + nested split regions** (drag a panel, drop zones highlight, region
  splits): this means **building a dock-tree layout manager from scratch** inside
  `camera.ui` — a recursive split-container model, drop-zone hit-testing, live
  preview overlays, per-region resize, tab groups. Every existing panel
  (`apply_layout` in 4 files) would be rewritten to render into a region rect handed
  down by the manager instead of anchoring itself. **Cost: 2–4+ weeks**, and it's a
  genuine sub-project with its own bug surface (this is the kind of thing that has
  eaten whole engine sprints elsewhere).
- **Floating panels as separate OS windows:** **effectively not feasible** on this
  stack. Panda3D single-window is the norm; multi-window/multi-`GraphicsWindow`
  support is fragile, and Ursina exposes none of it. Combined with the existing
  Panda3D-on-macOS resize quirks we already fight (Gotchas: lens aspect not
  auto-refreshed, `on_resize` dead), this is a "don't" — the cost/risk is wildly out
  of proportion to a solo game editor.

**Feasibility verdict, stated plainly so it's your call, not pre-filtered:** C1–C3
are clearly worth it and continuous with the layout code that already exists. C4 is
doable but the browser's fixed-bottom assumption makes it the first real "am I
building a UI framework?" step. **C5 drag-to-dock is technically buildable but is a
multi-week framework project**, and **C5 floating OS windows is not realistically
achievable** on Ursina/Panda3D. True Unity parity is *not* free and mostly *not*
worth it here — but the decision to skip it should be yours with these numbers in
hand, not made by omission.

## Suggested shape (for discussion — not a decision)
If the goal is "biggest feel improvement per unit effort," the cluster that lands
under ~2 days total and hits Ivan's stated priority order:
**B1 (cursor-pinned drag + hover) + B2-field (rotation numeric row) + C1 (collapsible
panels) + C2 (presets).** Rotation *ring* gizmo (B2-ring) and resizable panels (C3)
are the natural "if there's appetite for more" tier. C4/C5 are explicitly parked
until Ivan says the editor's UX is worth framework-scale investment.
