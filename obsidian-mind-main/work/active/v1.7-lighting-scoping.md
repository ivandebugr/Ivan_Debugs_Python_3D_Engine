# v1.7 Lighting Scoping — TF2-Style Look Under GLSL 1.20

> **Mode:** Scoping only — options with tradeoffs, no winner picked. Mirrors the
> [[v1.7-editor-ux-scoping]] pattern. This is the *input* to Ivan's decision, not the decision.
>
> **Hard constraint (settled, not revisitable):** OpenGL 2.1 / GLSL 1.20 compat profile —
> see [[brain/Key Decisions]] "v1.7 lighting stays on the 2.1/GLSL 1.20 compat profile"
> (2026-07-16). Every candidate below is expressible at `#version 120`.

Related: [[brain/Key Decisions]] · [[brain/Gotchas]] · [[brain/North Star]] · [[v1.7-editor-ux-scoping]]

## Part A — Current state (grounded in the code, not assumptions)

- **`Scripts/lit_shader.py`** — hand-written GLSL 1.20: ambient (`p3d_LightModel.ambient`
  + a 0.18 `ambient_boost` floor) + **one** light (`p3d_LightSource[0]`, handles both
  directional `w==0` and point `w==1` with attenuation), flat Lambert `max(N·L, 0)`.
  No shadows, no specular, no rim. Verified clean on the real 2.1 context via
  `tools/shader_harness.py`. Deliberately **outside** `compat.py`'s patch loop.
- **Who is lit:** level blocks (`main.py:395`) and ground (`main.py:514`) only.
  Enemies, the weapon viewmodel, and pickups render **unlit** — the near-black-MTL
  "black guns" issue (documented in the lit_shader docstring) is why characters were
  never put on the lit path.
- **Shadows today:** raycast blob quads via `Scripts/ground_shadow.py` (`GroundShadow`) —
  player (`player_controller.py:64`) and enemies (`enemy.py:126`). One downward raycast
  per owner per frame, soft-edged by texture. No world-geometry shadows at all.
- **Scene lighting lifecycle:** sun + ambient are created inside `main_menu()` *after*
  its scene sweep. Anything that adds GPU-side light state (shadow buffers) must survive
  the `return_to_menu()` → `_clear_gameplay_entities()` teardown path.
- **Proven on this stack:** offscreen buffers work on this GL context
  (`tools/retina_harness.py` exercised them); Ursina entity tints go through
  `setColorScale` → `p3d_ColorScale`, so the `p3d_Color` **vertex-color channel is free**
  on generated geometry (level cubes) for baked data.

## Part B — What "the TF2 look" actually decomposes into

From Valve's NPAR 2007 paper (*Illustrative Rendering in Team Fortress 2*), ranked by
what's engine work vs. what isn't:

| Ingredient                                                        | Nature                             | Cheap under GLSL 120?   |
| ----------------------------------------------------------------- | ---------------------------------- | ----------------------- |
| Half-Lambert warped diffuse (soft terminator, no black backsides) | fragment math                      | ✅ trivial               |
| Warm/saturated grade (warm sun, cool ambient)                     | scene tuning + shader math         | ✅ trivial               |
| Rim highlight + simple Phong specular on characters               | fragment math + character wiring   | ✅ cheap                 |
| Soft-edged shadows                                                | blob/baked **or** shadow map + PCF | ⚠️ this is the fork     |
| Painterly textures                                                | **asset work, not engine**         | n/a — out of scope here |

The signature TF2 read is mostly items 1–3 plus art direction. Real shadow mapping is
the only ingredient that needs new render-pass machinery.

## Candidate L1 — Stylized-sun shader upgrade (Half-Lambert + rim + spec + warm grade)

**What:** extend `lit_shader.py` in place: replace `max(N·L,0)` with Half-Lambert
(`pow(NdotL*0.5+0.5, 2.0)` or a smoothstep ramp — no ramp texture needed), add a
view-dependent rim term (`pow(1-N·V, k)` — `eye_position` varying already exists), a
Phong specular lobe, and an in-shader warm/cool grade. Put the lit shader on enemy
models (and optionally the viewmodel) so characters get the rim/spec treatment.

- **Requires:** fragment/vertex edits only — **no new render passes, no new light types,
  no new textures.** A handful of new uniforms with `default_input` so existing call
  sites don't change. Wiring `shader=lit_shader` into `enemy.py`.
- **Breaks / revisit:** the 0.18 `ambient_boost` floor double-lifts once Half-Lambert
  is in (HL already rescues dark backsides) — retune or drop it. Character MTLs with
  near-black Kd need a look, though HL itself largely defuses that. Re-verify via
  `tools/shader_harness.py` (same harness, same pattern — stays outside compat's loop).
- **Perf (M3):** a few dozen ALU ops/fragment — unmeasurable at this resolution.
  Ceiling unchanged from today; effectively free.
- **Risk: LOW.** Effort ≈ 1 session shader + 1 session tuning/wiring. The cost is art
  tuning, not code.

## Candidate L2 — Soft-shadow illusion without shadow mapping (blobs everywhere + baked vertex AO)

**What:** double down on the machinery that exists. (a) `GroundShadow` already gives
soft contact shadows for player/enemies — extend to any dynamic prop that needs
grounding. (b) For static world geometry, CPU-bake ambient occlusion / corner
darkening at `load_level()` time into **vertex colors** — `lit_shader` already
multiplies `vertex_color`, so the shader needs *zero* changes; level cubes are
generated meshes with a free `p3d_Color` channel (confirmed: Ursina tints ride
`p3d_ColorScale`, not vertex color).

- **Requires:** no render passes, no shader edits. A bake step at level load (rays
  against level geometry, O(verts × samples) — sub-second for current block counts),
  optional cache sidecar. Editor needs a re-bake trigger after block edits (cheap).
- **Breaks / revisit:** loaded OBJ models may carry their own vertex colors — bake
  applies to generated level geometry only (fine; that's where AO reads anyway).
  Level-load time grows slightly. `raycast(ignore=...)` rules apply to the bake
  (Hard Constraint #2 — instances, never classes).
- **Perf (M3):** zero added per-frame GPU cost; blob raycasts are the already-paid
  pattern (1/owner/frame). Ceiling unchanged.
- **Risk: LOW–MEDIUM** (bake correctness/seams is the tuning surface). Effort ≈ 1–2
  sessions. Gets the "soft-edged, painterly-grounded" read without any new GL surface.

## Candidate L3 — Real shadow mapping for the sun (single directional + PCF)

**What:** the only candidate with new render-pass machinery. Panda3D 1.10 has this
built in: `sun._light.setShadowCaster(True, 1024, 1024)` makes Panda create the depth
buffer + light camera and supply `p3d_LightSource[0].shadowViewMatrix` and the shadow
sampler to the shader automatically. GLSL 1.20 expresses the sampling side natively
(`sampler2DShadow` + `shadow2D()` are core since 1.10). Ursina's stock
`lit_with_shadows_shader` is exactly this shape at `#version 150` — the work is a
down-port of that pattern into our `#version 120` shader: shadow-coord transform in
the vertex stage, 4–16-tap PCF `shadow2D` compare in the fragment stage, bias tuning.

- **Requires:** depth-render pass (Panda-managed offscreen buffer), new shader inputs
  (shadow matrix + sampler in the LightSource struct), bias/acne/peter-panning tuning.
  **Must be spiked through `tools/shader_harness.py` + a windowed test first** —
  offscreen color buffers are proven on this stack (retina harness), but a
  *depth-format FBO* on Apple's deprecated GL 2.1 driver is precisely the kind of
  driver-dependent behavior this project has been burned by ([[brain/Gotchas]]).
- **Breaks / revisit:**
  - `lit_shader.py` roughly doubles its surface — the flat version took a full session
    to get right; budget 2–3 sessions including the harness spike and teardown testing.
  - **Teardown lifecycle:** sun is recreated in `main_menu()` after the sweep; the
    shadow buffer must be released/recreated cleanly across `return_to_menu()`. This
    project's history says teardown is where the bodies are buried (bullet pool,
    NodePath assertions) — treat as a first-class test target, not an afterthought.
  - `GroundShadow` blobs become partially redundant — decide keep (contact shadows
    inside shaded regions) or remove per-owner.
  - Editor viewport renders sun-lit geometry too — the depth pass runs there as well;
    verify editor perf/visuals.
- **Perf (M3):** scene geometry drawn twice (depth pass) — current content (hundreds
  of cubes, ≤ tens of enemies) is a trivial vertex load; expect ≲1 ms added at a
  1024²–2048² map. 4–16-tap PCF at window resolution is well inside M3 fill rate.
  Realistic ceiling stays 120 fps for current content — **the risk is driver/FBO
  behavior on the deprecated GL stack, not throughput.**
- **Risk: MEDIUM–HIGH** (macOS GL 2.1 FBO unknowns + teardown lifecycle). Highest
  reward: real, moving, soft-edged world shadows. **No new light types** — sun only.
- **Softness note:** TF2-soft edges need a decent PCF kernel (or bigger map). 4-tap
  reads "soft-ish"; 9–16-tap reads painterly and is still cheap here.

## Candidate L4 — Local point/spot lights (multi-light loop) for warm interior mood

**What:** generalize `p3d_LightSource[0]` to a fixed-size GLSL 120 uniform array
(`p3d_LightSource[N]`, compile-time N ≈ 4) with a per-fragment loop. The point-light
math (w==1 branch + attenuation) already exists in the shader — the loop is mostly
mechanical; spotlights add cone params (`spotDirection`/`spotCosCutoff` live in the
same Panda struct). Panda fills array slots for lights attached to render, sorted by
priority.

- **Requires (engine half, ~1 session):** shader loop + N constant; a placeable light
  entity class with clean creation/teardown through `_clear_gameplay_entities()`.
- **Requires (product half — the real cost, 2–4 sessions):** ⚠️ **this is the
  candidate with the direct editor-tooling dependency.** Placing/moving/coloring
  lights needs: a `level.json` schema extension, a gizmo-selectable proxy entity in
  the editor, inspector fields (color/range/intensity), undo/redo integration, and
  serialization — a feature on the scale of a [[v1.7-editor-ux-scoping]] work item,
  not a shader tweak.
- **Breaks / revisit:** every single-light assumption in `lit_shader.py` (struct →
  array, ambient retune). No shadows from these lights — shadowed point lights are
  out of budget on this stack, full stop.
- **Perf (M3):** fragment cost scales ×N on lit geometry; N=4 is fine at window
  resolution. Ceiling unchanged in practice.
- **Risk:** engine MEDIUM-LOW; product scope MEDIUM-HIGH (editor tooling dominates).

## Summary matrix

|                          | New passes | New light types | Editor dependency                | Perf hit (M3) | Risk        | Effort                   |
| ------------------------ | ---------- | --------------- | -------------------------------- | ------------- | ----------- | ------------------------ |
| **L1** stylized sun      | none       | none            | none                             | ~0            | LOW         | 1–2 sessions             |
| **L2** blobs + baked AO  | none       | none            | re-bake trigger (small)          | ~0            | LOW–MED     | 1–2 sessions             |
| **L3** shadow map + PCF  | depth pass | none            | none (verify editor viewport)    | ≲1 ms/frame   | MED–HIGH    | 2–3 sessions incl. spike |
| **L4** point/spot lights | none       | **yes**         | **yes — full placement tooling** | small ×N      | MED (scope) | 1 + 2–4 sessions         |

**Composition notes (not a recommendation):** L1+L2 is the "TF2-ish in a week, zero
new GL surface" bundle and is prerequisite-free. L3 supersedes the baked-AO half of
L2 but not the contact blobs. L4 is orthogonal mood work gated on editor tooling.
Painterly textures — the single biggest TF2 ingredient — are asset work and appear in
no candidate.

**Decision status:** none picked. Next step is Ivan's call; if L3 is in play, the
depth-FBO harness spike is the mandatory first gate (report-back before any shader
writing, per the supersampling-spike pattern in [[brain/Key Decisions]]).
