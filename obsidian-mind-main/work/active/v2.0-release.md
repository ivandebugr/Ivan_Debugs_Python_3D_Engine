# v2.0 — Public Release

Blocked on: [[work/active/v1.5-gameplay-systems]]. See [[work/archive/version-map]] for sequencing.

v2.0 is the first public release milestone. Four systems ship together. Each is independent enough to implement in order within a single long session per system.

## System A — Python Script Modding

### Two-layer architecture

#### Layer 1 — `mod.json` (declarative, no Python required)

```json
{
  "mod_id": "example_mod",
  "version": "1.0",
  "author": "your_name",
  "enemy_types": [
    {
      "id": "fast_enemy",
      "display_name": "Fast Enemy",
      "model": "cube",
      "scale": [1, 2, 1],
      "hp": 50,
      "speed": 12,
      "shoot_cooldown": 0.5,
      "detection_range": 120,
      "attack_range": 40,
      "behaviour": "aggressive"
    }
  ],
  "block_types": [
    {
      "id": "metal_block",
      "display_name": "Metal Block",
      "model": "cube",
      "texture": "assets/textures/metal.png",
      "collision": true
    }
  ],
  "weapon_types": [
    {
      "id": "fast_gun",
      "display_name": "Fast Gun",
      "damage": 15,
      "fire_rate": 0.08,
      "bullet_speed": 70,
      "bullet_color": [0.2, 1.0, 0.2]
    }
  ]
}
```

The `"behaviour"` field uses v1.4 preset names (see [[work/archive/2026/v1.4-enemy-behaviour-trees#Layer 3 — Tree configuration]]). A `mod.json` alone (no `hooks.py`) is a valid, complete mod — no Python required.

#### Layer 2 — `hooks.py` (optional Python lifecycle callbacks)

All engine access goes through `ModAPI`. Hooks that directly mutate internal state outside `ModAPI` are unsupported and may break between versions.

**Hook catalogue:**

`on_mod_load(api)` — fired at engine startup, after `ModLoader.load_all()`. Use to register custom types, load assets via `api.load_texture()` / `api.load_model()`, and set up module-level state for the mod. **Allowed:** register types, load assets. **Disallowed:** access `api.player` (player doesn't exist yet), call `api.spawn_enemy()`.

`on_game_start(api)` — fired after `game.start()` initializes the player and level. Use to read `api.player` position, `api.enemies` list, or spawn extra entities. **Allowed:** `api.spawn_enemy()`, `api.player` read access, `api.show_notification()`. **Disallowed:** modify player HP directly (use `api.player.health` property only).

`on_enemy_spawn(api, enemy)` — fired after an Enemy is created and registered. Read state only — the enemy is not yet in its first `update()`. **Allowed:** read `enemy.position`, `enemy.health`. **Disallowed:** `enemy.die()` (enemy has not lived yet — call `api.spawn_enemy()` with a different type instead if you need substitution).

`on_enemy_death(api, enemy)` — fired inside `enemy.on_die()`, before `destroy()` is deferred. Use to spawn loot, trigger effects, chain spawns. **Allowed:** `api.spawn_enemy()`, `api.play_sound()`. **Disallowed:** `enemy._alive = True` (cannot resurrect via private field); use `api.spawn_enemy()` at the enemy's position instead.

`on_player_damage(api, amount, source) -> float | None` — fired before damage is applied. Return a `float` to override the damage amount (return `0.0` for invincibility, `amount * 2` for double damage). Return `None` to apply the original amount. Only the last hook's return value is used if multiple mods hook this event — hooks are called in load order.

`on_player_death(api)` — fired before position reset. **Disallowed:** `api.player.die()` — player death is already in progress.

`on_level_load(api, level_data: list[dict]) -> list[dict]` — fired after `level.json` is parsed, before entities are created. Mutate and return the list to add/remove/modify entries. Return `level_data` unchanged if no modification needed. **Allowed:** add entries, modify existing entries. **Disallowed:** import from `main.py` inside this hook.

`on_update(api, dt: float)` — fired every frame during gameplay (`game.state == Game.PLAYING`). Keep this hook lightweight. **Disallowed:** `raycast()` calls (budget is already spent by player movement and bullets); add per-enemy logic to behaviour nodes instead (see [[work/archive/2026/v1.4-enemy-behaviour-trees]]).

### `ModAPI` class (stub with full surface)

```python
class ModAPI:
    # Read-only game state
    @property
    def player(self) -> Player: ...          # current player; None if not in PLAYING state
    @property
    def enemies(self) -> list[Enemy]: ...    # live enemies only (alive == True)
    @property
    def current_level_data(self) -> list[dict]: ...  # parsed level.json data

    # Actions
    def spawn_enemy(self, position: Vec3, enemy_type: str = "default") -> Enemy: ...
    def register_enemy_type(self, type_def: dict) -> None: ...   # same schema as mod.json enemy_types entry
    def register_block_type(self, type_def: dict) -> None: ...   # same schema as mod.json block_types entry
    def register_weapon_type(self, type_def: dict) -> None: ...  # same schema as mod.json weapon_types entry
    def load_texture(self, path: str): ...   # returns Panda3D Texture handle
    def load_model(self, path: str): ...     # returns Panda3D NodePath
    def show_notification(self, text: str, duration: float = 2.0) -> None: ...
    def play_sound(self, path: str, volume: float = 1.0) -> None: ...
```

### `ModLoader` (`Scripts/mod_loader.py`)

```python
class ModLoader:
    def __init__(self, mods_dir: str = "mods/"): ...
    def load_all(self) -> None: ...          # scans mods_dir, calls load_mod() for each subfolder
    def load_mod(self, path: str) -> None:   # reads mod.json, optionally imports hooks.py
    def get_enemy_type(self, type_id: str) -> dict | None: ...
    def get_block_type(self, type_id: str) -> dict | None: ...
    def get_weapon_type(self, type_id: str) -> dict | None: ...
    def call_hook(self, name: str, *args) -> list: ...  # calls hook on all loaded mods in load order
    def unload_all(self) -> None: ...        # called by return_to_main_menu()

mod_loader = ModLoader()  # module-level singleton
```

A failing mod must not crash the engine — every `load_mod()` and every `call_hook()` wraps in `try/except Exception` and logs to the status bar. The engine continues without that mod.

### `level.json` mod schema additions

```json
{"type": "enemy", "position": [...], "enemy_type_id": "fast_enemy"},
{"type": "block",  "position": [...], "block_type_id":  "metal_block"}
```

Both fields are optional and backwards-compatible. Absent `enemy_type_id` → built-in `"default"` enemy. Absent `block_type_id` → built-in cube block.

### Level editor palette integration

After `ModLoader.load_all()`, `LevelEditor` queries `mod_loader` for all registered enemy types and block types. The entity palette panel (in the editor sidebar) shows built-in types and mod-registered types together, with display names from `mod.json`. The editor does not require mods to be active to edit levels that use mod-registered types — it just places the `enemy_type_id` / `block_type_id` string in `level.json` and relies on the runtime to resolve it.

### Security model

v2.0 ships a trust-based modding model — mods are Python code that runs in the same process as the engine with no restrictions beyond the `ModAPI` surface convention. Before loading, `ModLoader` performs a shallow AST walk (`ast.walk`) on `hooks.py` to scan for direct imports of `main.py`, `collision_system.py`, or internal engine modules; it logs a warning if found but does not block loading. This is a courtesy warning, not a sandbox. `RestrictedPython` or `subprocess` isolation is explicitly deferred to a post-v2.0 milestone (see [[work/archive/version-map#Post-v2.0 Parking Lot]]). Modders are expected to read the `ModAPI` documentation and follow the hook contract — the engine does not enforce it at runtime beyond the try/except wrapper.

### First mod worked example: `mods/example_mod/`

This is the end-to-end verification reference for the modding system. `mod.json` defines a `fast_enemy` type (hp=50, speed=12, behaviour="aggressive"). `hooks.py` implements `on_enemy_death`: when any enemy dies, it spawns two `fast_enemy` entities at random positions within 5 units of the dead enemy's position, creating a "splits on death" mechanic. The example mod ships in the repo at `mods/example_mod/` and is the first thing to test when the modding system is implemented.

## System B — Packaged Runtime

### PyInstaller (primary), Nuitka (stretch goal)

`ivans-3d-engine.spec` is committed to the repo. One-folder output (`--onedir`, not `--onefile`) is used so modders can drop their `mods/` subfolder into the distribution directory — a single-file executable cannot be extended.

Key spec sections:
```python
datas = [
    ('assets/', 'assets/'),
    ('mods/', 'mods/'),
    ('level.json', '.'),
    ('Scripts/', 'Scripts/'),
    # Ursina data files resolved via ursina.__file__
    # Panda3D .so / .dylib resolved via panda3d.__file__

hiddenimports = [
    'panda3d.core', 'panda3d_gltf',
    'ursina', 'ursina.shaders', 'ursina.prefabs',
    # Add as discovered during test build

excludes = ['pytest', 'sphinx', 'mypy', 'black', 'ruff']
```

Platform targets: macOS (primary — all development is macOS), Windows (secondary — tested in CI), Linux (community-supported — not CI-gated in v2.0).

### GitHub Actions `build.yml`

Triggered on tag push matching `v2.*`. Runs on `macos-latest` and `windows-latest` runners. Steps: checkout → install Python 3.10 → `pip install -r requirements.txt pyinstaller` → `pyinstaller ivans-3d-engine.spec` → upload `dist/` folder as a release asset attached to the tag. Linux build is a community contribution target, not CI-gated.

## System C — Gamepad Input Layer

### `Scripts/input_manager.py`

`InputManager` wraps both keyboard/mouse and gamepad behind a unified interface. The player controller calls `InputManager` methods; the underlying device is transparent.

```python
class InputManager:
    def get_move_vector(self) -> Vec2: ...   # WASD or left stick
    def get_look_delta(self) -> Vec2: ...    # mouse delta or right stick
    def is_pressed(self, action: str) -> bool: ...
    def on_action_down(self, action: str, callback) -> None: ...

# Actions:
# "shoot", "jump", "pause", "switch_weapon_next", "switch_weapon_prev", "interact", "reload"
```

Panda3D gamepad API:
```python
devices = base.devices.getDevices(InputDevice.DeviceClass.gamepad)
device = devices[0] if devices else None
left_x  = device.findAxis(InputDevice.Axis.left_x).value   # -1.0 to 1.0
left_y  = device.findAxis(InputDevice.Axis.left_y).value
right_x = device.findAxis(InputDevice.Axis.right_x).value
right_y = device.findAxis(InputDevice.Axis.right_y).value
a_pressed = device.findButton(GamepadButton.face_a).pressed  # A / Cross
```

Default bindings: left stick = move, right stick = look (with sensitivity scalar), A/Cross = jump, RB/R1 = shoot, Start = pause. Dead zone: 0.15 — axis values below this magnitude are treated as zero. Sensitivity configurable in `editor_prefs.json` as `"gamepad_sensitivity": 1.0`.

`Player.update()` replaces all `held_keys['w']` / `held_keys['s']` / etc. calls with `input_manager.get_move_vector()`. The keyboard path remains as fallthrough — when no gamepad is connected, `InputManager.get_move_vector()` reads `held_keys` directly, so zero behaviour change for keyboard/mouse players.

Module-level singleton: `input_manager = InputManager()` — initialized in `main.py` after `Ursina()` and before any player creation.

## System D — Procedural Level Generator

### `Scripts/level_generator.py`

`LevelGenerator.generate(seed, config) -> list[dict]` is a **pure Python function** — no Ursina import, no Panda3D import, no global state. It takes a seed and config dict and returns a valid `level.json` format list. The level editor calls it, then loads the result normally.

```python
config = {
    "seed": 42,
    "width": 40,
    "depth": 40,
    "min_room_size": 5,
    "max_depth": 4,
    "enemy_count": 5,
    "block_texture": "white_cube.png",
    "wall_height": 3
}
output = LevelGenerator.generate(42, config)
# output is a list[dict] — identical format to level.json contents
```

### BSP room-and-corridor algorithm

1. Start with a bounding rectangle `(0, 0, width, depth)`
2. Recursively split into two sub-rectangles (alternating horizontal/vertical) until leaf rectangles are smaller than `min_room_size × 2` or `max_depth` is reached
3. Place a room (floor + walls) in each leaf rectangle, sized randomly within the leaf bounds
4. At each split point, carve a corridor connecting the two child rooms (L-shaped, 1 unit wide)
5. Distribute `enemy_count` enemies in randomly chosen room interiors, at least 3 units from walls
6. Place a `"checkpoint"` trigger zone near the player spawn (origin corner) and a `"win_condition"` trigger zone at the far corner

Output uses all v1.2 schema fields (`colour`, `rotation`) and all v1.3 schema fields (`model`) — fully schema-compatible. The generator itself chooses colours and rotations to vary the visual feel (e.g., wall blocks at rotation `[0, 0, 0]`, ceiling blocks at `[0, 0, 180]`).

The generator is deterministic — same seed + same config always produces the same output. This is a design requirement, not an accident: level seeds can be shared between players.

### Level editor "Generate" button

A "Generate" button in the level editor toolbar opens a small config panel (seed field, width/depth sliders, enemy count). On confirm:
1. Call `LevelGenerator.generate(seed, config)`
2. Load result into `LevelEditor.blocks` and `LevelEditor.enemies` exactly as if the user had placed them
3. Clear the undo stack — generation is a ground-truth reset point
4. Hand-edit and save normally via the existing save workflow

## Implementation Order

1. `ModLoader` + `mod.json` parsing (no hooks yet; just type registration)
2. `level.json` `enemy_type_id` / `block_type_id` support in `load_level()` and `load_existing_level()`
3. Level editor mod-type palette (entity palette shows built-in + mod types)
4. `ModAPI` stub + `on_mod_load` hook
5. Remaining hooks in game-lifecycle order: `on_game_start` → `on_enemy_spawn` → `on_enemy_death` → `on_player_damage` → `on_player_death` → `on_level_load` → `on_update`
6. AST import scan warning in `load_mod()`
7. `ModLoader.unload_all()` in `game.return_to_menu()` → `_clear_gameplay_entities()`
8. `InputManager` + keyboard fallthrough (zero behaviour change for keyboard users — mandatory before gamepad)
9. Gamepad device detection + axis/button polling
10. `Player.update()` integration (swap `held_keys` → `input_manager`)
11. `LevelGenerator` BSP algorithm (pure Python, fully testable without Ursina)
12. Level editor Generate button + config panel
13. PyInstaller `.spec` + local build test on macOS
14. GitHub Actions `build.yml` — macOS + Windows runners

## Constraints

- No fourth collision authority — `TriggerZone` (from v1.5) uses `intersects()`; `AmmoPickup` uses `intersects()`; no new raycast authority
- Mod hooks call `api.enemy.die()` only via `ModAPI` — direct access to `enemy._alive` or `collision_system.unregister()` is unsupported
- `InputManager` must not break the keyboard path — keyboard/mouse players must notice zero difference
- `LevelGenerator` is pure Python with no Ursina or Panda3D imports — testable standalone with `python -c "from Scripts.level_generator import LevelGenerator; ..."`
- PyInstaller spec excludes dev files: `tests/`, `*.spec` itself is not bundled, `.git/` excluded automatically
