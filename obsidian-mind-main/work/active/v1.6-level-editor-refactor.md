# v1.6 — Level Editor Refactor

Blocked on: [[work/active/v1.3-asset-import-pipeline]], v1.4, and v1.5 shipping first — AND a manual system-design review of the other scripts that hasn't happened yet. See [[work/archive/version-map]] for sequencing.

## Goal

Split `level_editor.py` (currently 4000+ lines) into smaller, navigable modules. Two motivations, both explicit:

1. **Human navigability** — a single 4k+ line file is hard to hold in your head or scan for a specific feature.
2. **Agent productivity** — smaller files per concern mean less context an agent needs to load to work on any one task, which directly affects how effectively Claude Code can operate on the editor going forward.

This is deliberately sequenced **after** v1.3–v1.5 feature work, not interleaved with it — the editor keeps absorbing new features (asset pipeline, gameplay systems) and refactoring mid-feature-stream would just create merge friction against a moving target. It is also gated on a manual design review the user is doing personally, not on an agent-proposed breakdown.

## Open Questions

Nothing below is decided — this note exists to mark the milestone, not to answer these:

- Where do module boundaries fall?
- Does browser-panel / inspector / gizmo / undo-redo-wiring code naturally separate, or does it stay coupled in ways that resist a clean split?
- Is this a mechanical extract-without-behavior-change pass, or does it end up touching logic?
- What regression-testing approach confirms no behavior changed during the split?
