# v1.7 Particle System Scoping — Weapon FX / Death FX / Ambience Under GLSL 1.20

> **Mode:** Scoping only — options with tradeoffs, no winner picked. Mirrors the
> [[v1.7-bloom-scoping]] / [[v1.7-lighting-scoping]] pattern. This is the *input* to
> Ivan's decision, not the decision.
>
> **Hard constraint (settled):** OpenGL 2.1 / GLSL 1.20 compat profile. Every candidate
> below is expressible at `#version 120` or via fixed-function.

Related: [[v1.7-bloom-scoping]] · [[v1.7-lighting-scoping]] · [[brain/Gotchas]] · [[brain/Key Decisions]] · [[brain/North Star]]

Covers three effect families:
1. **Weapon FX** — shell casings, impact sparks (muzzle flash already ships).
2. **Death / destruction** — enemy death burst, explosions. (Today `Enemy.on_die()` destroys
   the health bar and blob shadow and the enemy just vanishes — zero visual feedback.)
3. **Environmental ambience** — dust motes, atmosphere. Unlike 1–2 these are *persistent*,
   not burst-shaped — different perf profile, see Part C.

---

## Part A — Current state (grounded in the code, verified today, not assumed)

**The pooled-cosmetic pattern already ships and is battle-tested.** `MuzzleFlashPool`
(`weapon.py:239-268`): 4 pooled billboard quads, round-robin (no free-list — flashes are too
short-lived to need one), position-parked at `y = -10000` (never `enabled`-toggled — Hard
Constraint 6, the `unstash()` assertion), lazy-built on first `play()`, with a **stale-drain
guard** (`is_empty()` check per acquire) so quads destroyed by `main_menu()`'s scene sweep are
replaced instead of crashing. Purely cosmetic: no collider, never registered with
`collision_manager`, never an authority in the damage path.

**Teardown is a three-path problem, but there is a single wiring point.** The paths:
1. `game.return_to_menu()` → `_clear_gameplay_entities()` (`main.py:451`), which calls
   `reset_bullet_pools()` (`weapon.py:718`) *first*, before any entity destruction.
2. `main_menu()`'s `scene.entities` sweep — handled defensively by each pool's stale-drain guard.
3. **F5 editor-playmode exit** (`editor_playmode.py:85-110`) — calls `_clear_gameplay_entities()`
   *plus* its own isinstance-based sweep for entity types the name-sweep misses.

A new pooled effect type that (a) funnels its `reset()` into `reset_bullet_pools()` (rename to
`reset_effect_pools()`) and (b) carries the stale-drain guard gets **all three paths for free**.
This is exactly the wiring the muzzle flash pool already proved. Anything that does *not* live
in `scene.entities` as an Ursina Entity (→ P4) gets none of this for free.

**Rendering / lighting substrate:**
- `lit_shader.py` — project-owned `#version 120` lit path (Half-Lambert + additive spec/rim),
  verified via `tools/shader_harness.py` on GL 2.1 Metal. Deliberately outside `compat.py`'s
  patch loop. Custom `#version 120` shaders with per-vertex attributes are therefore a *proven*
  capability of this stack, with a headless compile-gate harness already in place.
- The **fixed-function no-shader path is also proven** on this context — it's the "unlit
  fallback" the black-guns bug rendered through for months. Relevant to P4's renderer.
- `panda3d.physics` and `direct.particles.ParticleEffect` **import cleanly** on this install
  (Panda3D 1.10.16, verified 2026-07-17). P4 is real, not hypothetical.

**Existing frame budget already spent per frame:** swept player rays (5), per-bullet raycasts
(pools of 30+60), enemy behaviour trees, blob ground-shadows, the viewmodel second camera pass,
and (pending L3) a shadow FBO pass. Particles land on top of this, not on an empty frame.

## Part B — The lighting answer (shared by all candidates — settle this first)

**Particles render UNLIT. None of the four candidates should touch `lit_shader`.**

- **Sparks / muzzle / explosion cores → additive blend, depth-test ON, depth-write OFF.**
  Additive effects *are* light sources visually; running them through a lighting model
  double-dips (industry norm — additive FX are unlit in every mainstream engine). Depth-write
  off is mandatory: additive quads writing depth punch holes in each other.
- **Smoke / dust → alpha blend, unlit, ambient-tinted at spawn.** A camera-facing billboard has
  no meaningful normal (it always faces the viewer), so Half-Lambert on it evaluates to a
  constant — running `lit_shader` would cost a full lighting model to compute a flat tint.
  Instead multiply the particle color by the scene ambient / sun color **once on the CPU at
  spawn**. One multiply, visually equivalent.
- **Shell casings are the only defensible `lit_shader` users** — they're real tiny meshes lying
  in the world, not billboards. But at ~5 cm on screen, a constant warm tint is visually
  indistinguishable from the full lit path. Optional either way; not a system-shaping decision.

Consequence: the particle rendering question is *independent* of the lighting/L-track work.
No new lit-shader variants, no `p3d_LightSource` plumbing.

## Part C — Shared perf model (Apple M3, 60 fps target)

Realistic worst-case firefight, simultaneous live particles:

| Source | Estimate |
|---|---|
| Impact sparks (full-auto, ~6/impact, 0.4 s life) | ~25 |
| Shell casings (1/shot, 2–3 s life) | ~25 |
| 2 enemy deaths overlapping (~20 each, ~1 s) | ~40 |
| Muzzle flashes (existing pool) | 4 |
| **Burst subtotal** | **~95, spikes to ~150–200** |
| Ambient dust (persistent, whole level) | +50–150 |

Two distinct cost regimes on this stack:
- **Entity-per-particle** (P1/P2): each particle is an Ursina Entity → one Geom → **one draw
  call**, plus a Python transform write per frame. The ceiling is Python loop cost + draw-call
  count, not the GPU. Realistic ceiling on M3 under this game's existing frame load:
  **~200–400 simultaneous** before frame time visibly moves. Burst FX fit; **persistent ambient
  dust is what pushes it over.**
- **Batched geometry** (P3) / **C++ sim** (P4): one draw call per emitter / per effect.
  **Thousands** of particles before anything registers; the new bottleneck is fill rate
  (large overlapping additive quads) — keep quads small, cap overdraw, not count.

Rule of thumb: burst effects alone → entity-per-particle is fine. Ambient dust as a headline
feature → you want a batched path for *at least* the dust.

## Candidate P1 — Generalize `MuzzleFlashPool` → one `ParticlePool` (entity-per-particle, CPU sim)

One pool class, one particle Entity type with per-particle state (velocity, gravity flag,
lifetime, fade/scale curve, optional floor-stop). Emission helpers like
`burst(position, count, preset)`. Casings are just particles with `gravity=True` and a
floor-stop (cache floor `y` at spawn from one raycast — no per-frame raycasts).

- **Requires:** ~150–250 lines in a new `Scripts/particles.py`. One manager-driven update loop
  iterating live particles (a single `update()` owner, *not* per-particle `update()` methods —
  avoids Ursina's per-entity dispatch overhead). Preset dicts per effect (spark/smoke/casing/dust).
- **Reused vs new:** Reuses everything the muzzle flash proved — parking, lazy build,
  stale-drain guard, `reset()` wiring, `billboard=True`, `_resolve_texture`. New: the sim loop
  and presets. `MuzzleFlashPool` itself can stay untouched (zero regression risk) or be folded
  in later.
- **Teardown:** `reset()` registered in `reset_bullet_pools()` → all three paths covered by
  the one existing wiring point. Identical shape to what already survives return-to-menu and F5.
- **Perf ceiling:** ~200–400 live particles. Fine for all burst FX; ambient dust must be
  budgeted (e.g. ≤50 motes near the player) or cut.
- **Risk:** lowest of the four. The known failure mode is scope creep inside "one system does
  physics *and* visuals" — the per-particle branch (`if gravity: … if floor_stop: …`) is where
  it silently grows.

## Candidate P2 — Two lighter systems on a shared pool base (physics debris vs pure billboards)

Extract the pooling mechanics (park / lazy build / drain guard / reset) into a small
`EffectPool` base. Two subclasses: **`DebrisPool`** (shell casings, chunky gibs — real small
meshes, velocity + gravity + spin + floor stop, pool ≈ 16–32) and **`BillboardFXPool`**
(sparks/smoke/dust — lifetime + drift + fade, **zero physics code**, pool ≈ 64–128).
`MuzzleFlashPool` becomes the third (trivial) subclass or stays independent.

- **Requires:** same total line count as P1, split differently. The design honesty win: the
  billboard system never grows physics branches, the debris system never grows fade-curve
  logic. Matches the real fork in the requirements (casings want a little physics; nothing
  else does).
- **Reused vs new:** same reuse as P1 plus the base-class extraction. If `MuzzleFlashPool` is
  rebased onto the base, that's the one regression-risk touch to shipped code.
- **Teardown:** one `reset_effect_pools()` aggregator resets all subclass pools — still a
  single wiring point into the three paths. *Discipline note:* every future `EffectPool`
  subclass must register with the aggregator at construction (auto-register in
  `EffectPool.__init__`, don't rely on remembering — this project's HealthBar/light_sun
  lesson generalized).
- **Perf ceiling:** identical to P1 (~200–400) — still entity-per-particle. The win is code
  shape, not throughput. Same ambient-dust caveat.
- **Risk:** low. Slightly more up-front structure than P1; pays off only if the two systems
  genuinely diverge (they likely do).

## Candidate P3 — Batched single-Geom emitters (`#version 120` stateless GPU particles) + tiny debris pool

One Ursina Entity per *emitter*, holding a single Geom of N pre-allocated quads. At burst time,
write per-vertex spawn attributes once (origin, velocity, birth time, seed); a `#version 120`
vertex shader computes `p = p0 + v·t + ½g·t²`, billboards the corner from the view matrix, and
fades by age. **Zero per-frame CPU work per particle; one draw call per emitter.** GLSL 1.20
handles custom attributes/uniforms fine — the same capability class `lit_shader.py` already
proved, gated by the existing `tools/shader_harness.py`.

- **Requires:** the most new machinery: GeomVertexData plumbing (below Ursina's Mesh
  abstraction, or via Mesh with a custom vertex format), one new ~60-line shader pair, a
  ring-buffer of quad slots per emitter. **Casings cannot live here** (no per-particle
  collision/floor-stop on the GPU) — P3 in practice is *P3 for billboards + a P1/P2-style
  16-slot debris pool for casings*. That hybrid should be stated up front, not discovered late.
- **Reused vs new:** reuses the shader-harness gate, teardown wiring (emitters are ordinary
  Ursina Entities → pool-reset registration works unchanged), texture pipeline. New: vertex
  format, shader, slot ring-buffer — the highest-unknown code in any candidate.
- **Teardown:** *simpler* than P1/P2 at runtime — a handful of emitter entities instead of
  hundreds of quads; same single reset wiring point. Fewer objects that can leak.
- **Perf ceiling:** thousands of particles; ambient dust across the whole level becomes free.
  New bottleneck is additive-blend fill rate — cap quad screen size, not count.
- **Risk:** highest implementation unknowns (GeomVertexData under Ursina 8.3.0; attribute
  binding through Ursina's Shader wrapper at `#version 120`). Deserves a half-day spike gate
  like the L3 FBO probe before committing — same playbook as [[v1.7-bloom-scoping]].

## Candidate P4 — Panda3D built-in particle subsystem (`direct.particles` + `panda3d.physics`)

Use the engine's own 20-year-old particle stack: `base.enableParticles()` (starts the
PhysicsManager/ParticleManager tasks), `ParticleEffect` configured **programmatically**
(skip `.ptf` files — they're `exec`'d Python with ancient tooling) with
`SpriteParticleRenderer`, `SphereVolumeEmitter`/`PointEmitter`, `LinearVectorForce` gravity.

- **Requires:** ~100 lines of effect-config code, plus `enableParticles()` lifecycle management
  across menu↔game↔editor transitions. Verified importable today on 1.10.16.
- **Reused vs new:** reuses nothing from this project's patterns; the sim and renderer come
  from C++ for free. Renderer draws via fixed-function — the path already proven on this GL 2.1
  context — **but** interaction with `compat.py`'s shader patching and Ursina 8.3.0 render
  states is untested. Unknowns: transparency/color-scale state, render-bin behavior vs the
  viewmodel/UI display regions. Spike-gate territory, same as P3.
- **Teardown — the disqualifying-grade risk to weigh honestly:** `ParticleEffect` is a raw
  NodePath under `render`, **not an Ursina Entity**. It is invisible to `scene.entities`, to
  `_clear_gameplay_entities()`'s name sweep, *and* to the F5 isinstance sweep. Every live
  effect must be tracked in a module-level registry with explicit `.cleanup()` wired into all
  three paths by hand — and a forgotten one leaks **silently forever**, with no sweep ever
  catching it. This is precisely the light_sun / HealthBar class of bug, minus the safety nets
  that eventually caught those. Also: casings still aren't solved — `LinearVectorForce` gives
  gravity but no floor collision, so a P1/P2-style debris pool is needed anyway.
- **Perf ceiling:** C++ sim update + per-frame geometry regen in C++ — high hundreds to ~2k
  particles comfortably.
- **Risk:** integration risk concentrated on exactly the two axes this project has been burned
  by repeatedly: teardown lifecycle and macOS-GL rendering-state surprises. The "free" C++ sim
  is real, but it buys throughput this game may not need (see Part C) at the cost of a fourth
  lifecycle regime.

## Comparison

| | P1 one pool | P2 two systems, shared base | P3 batched GPU + debris pool | P4 Panda3D built-in |
|---|---|---|---|---|
| New code | ~200 lines | ~250 lines | ~400 lines + shader | ~100 lines + lifecycle glue |
| Reuses muzzle-flash pattern | fully | fully (extracted) | for debris + wiring | not at all |
| Casings (physics) | in-system flag | dedicated DebrisPool | needs side debris pool | needs side debris pool |
| Teardown | existing wiring point | existing wiring point + auto-register | existing wiring point, fewer objects | **manual registry, no sweep safety net** |
| Perf ceiling (live particles) | ~200–400 | ~200–400 | thousands (fill-rate-bound) | ~1–2k |
| Ambient dust as headline feature | budgeted/capped | budgeted/capped | free | comfortable |
| Unknowns needing a spike | none | none | GeomVertexData + attrib binding under Ursina | render-state interop + task lifecycle |
| Failure mode to watch | physics/visual branch creep | premature abstraction | shader debug time | silent leaks across menu transitions |

## Open questions for Ivan (these pick the winner, not taste)

1. **Is ambient dust a v1.7/v1.8 headline feature or a nice-to-have?** Headline → P3's batched
   path earns its cost. Nice-to-have at ≤50 motes → P1/P2 cover everything.
2. **Do casings need to *collide with level geometry* or just stop at the floor?** Floor-stop
   (cached spawn raycast) is cheap in any candidate; true wall bounces would push toward the
   swept-ray patterns and should probably just be cut (RICE).
3. **How much future FX ambition?** (fire, rain, tracers, gibs) More families → P2's split or
   P3's throughput amortize; one-off polish → P1.

**Not revisitable:** particles are unlit (Part B); any candidate must plug into the
`reset_bullet_pools()` wiring point and carry the stale-drain guard (Part A); Hard Constraint 6
(no `enabled` toggling on pooled entities) applies to every pooled quad.
