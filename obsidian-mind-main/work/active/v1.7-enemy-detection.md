# v1.7 — Enemy Detection (Line-of-Sight + Range Gating)

## Context

Every combat node (`ChaseNode`, `AttackNode`) gated purely on **distance** — no line-of-sight check anywhere in the tree, so enemies chased and shot the player straight through walls. Documented in [[Gotchas#patrol_then_attack is invisible unless the route is >60u from wherever the player stands — 2026-07-06]]; the only mitigation had been a level-design rule (patrol waypoints >60u from spawn). This closes it with real code.

## What shipped (`7643f19`)

- **`DetectPlayerNode`** (`Scripts/behaviour_nodes.py`) — a condition node that fronts the chase/attack `Sequence` of every preset. Range check (same true-length convention as Chase/Attack) + line-of-sight. Returns only SUCCESS/FAILURE, never RUNNING, so a lost target aborts the Sequence at child 0 and the `Selector` falls through to the preset's own non-combat fallback (Patrol/Idle) — preserving the stateless restart-from-child-0 convention (see [[Patterns]] / `behaviour_tree.py` ownership contract).
- **`Enemy.can_see_player()`** (`Scripts/enemy.py`) — thin LOS accessor wrapping the pre-existing `_is_occluded()` raycast (walls block, player does not). The node delegates the raycast here so `behaviour_nodes.py` stays Ursina-import-free — same delegation pattern as `chase_step`/`patrol_step`. (This method physically landed in the preceding walk-bob commit `27c51c7`, which swept the `enemy.py` working-tree edit; functionally correct there, left in place rather than rewriting another session's local-only commit.)
- Wired into all five presets; detector range pinned to each preset's own `ChaseNode` detection_range.

## Design decisions (resolved explicitly, per task)

1. **FleeNode bypasses detection.** The `flee_when_low` flee branch is intentionally **ungated** — a low-HP enemy flees in reaction to damage already taken, not to a fresh sighting, so it keeps fleeing even after rounding a corner and breaking LOS. Only that preset's chase/attack branch carries the gate.
2. **Hysteresis shipped from the start** (not deferred). `DETECT_LOSE_MARGIN = 1.15`: detection acquired at `detection_range`, held out to `detection_range * 1.15`, so a player hugging the boundary can't flicker frame-to-frame. LOS has **no** hysteresis — a wall dropping in loses the target immediately (instant lose-track, per v1 spec).
3. **v1 scope only:** range + LOS. No field-of-view cone, no last-known-position memory — both flagged as easy future additions, deliberately out of scope.

## Verification

- 100 behaviour-tree unit tests pass (new `TestDetectPlayerNode` + `TestDetectPlayerNodeInSequence`; updated all five preset shape assertions + real-`Enemy` integration test; blocked-LOS fall-through tick tests for `default` → Idle and `patrol_then_attack` → Patrol).
- Smoke harness — **all 4 scenarios PASS** (`load_and_close`, `win_then_r`, `gameover_then_r`, `editor_f5_roundtrip`). `gameover_then_r` was failing HARNESS_ERROR at first, but that was a **pre-existing harness regression, not this change** (verified by stashing the detection edits). It was then diagnosed and fixed in the same session: the v1.7 themed-button animation defers `start_game()` by 0.1s, so the harness's fixed `step(3)` after clicking Play didn't reach PLAYING; `start_real_game()` now steps until PLAYING. See [[Gotchas#Smoke harness `win_then_r`/`gameover_then_r` fail HARNESS_ERROR after the v1.7 menu redesign — 2026-07-07]].
