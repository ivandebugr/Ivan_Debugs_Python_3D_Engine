# v1.5 — Gameplay Systems

Blocked on: [[work/archive/2026/v1.4-enemy-behaviour-trees]]. See [[work/archive/version-map]] for sequencing.

## Goal

Two independent gameplay systems that ship in the same version. Both are data-driven via `level.json` — no Python required for common use cases.

1. **Trigger/zone system** — invisible AABB volumes that fire callbacks on player enter/exit. Built-in actions: open doors, set checkpoints, kill planes, play sounds, win condition.
2. **Weapon inventory API** — multi-weapon slots with switch animations, per-weapon ammo, and pickup entities. Pistol, Shotgun, and Rifle built-in; designed for mod extension in v2.0.

## System A — Trigger/Zone System (`Scripts/trigger_system.py`)

### `TriggerZone` class

`TriggerZone` extends `AliveEntity`. It is an invisible AABB volume — no model, no texture, only a `BoxCollider`. It is registered as `Layers.TRIGGER`, a new bitmask entry.

**Critical rule: `Layers.TRIGGER` is NOT a damage path and must never appear as a source or target in `COLLISION_MATRIX`.** It exists solely for enter/exit detection. The player's `_swept_blocked` must explicitly skip `Layers.TRIGGER` entities — triggers must not physically block movement.

```python
class TriggerZone(AliveEntity):
    def __init__(self, position, scale, on_enter=None, on_exit=None, **kwargs):
        super().__init__(position=position, scale=scale, collider='box', **kwargs)
        register(self, Layers.TRIGGER)
        self.on_enter_callbacks = on_enter or []
        self.on_exit_callbacks  = on_exit or []
        self._player_inside = False

    def update(self):
        if not self.alive:
            return
        inside = self.intersects(player).hit
        if inside and not self._player_inside:
            for cb in self.on_enter_callbacks:
                cb()
        elif not inside and self._player_inside:
            for cb in self.on_exit_callbacks:
                cb()
        self._player_inside = inside
```

### Built-in callback actions (wired from `level.json`)

| Action | What it does | `on_enter` or `on_exit` |
|---|---|---|
| `"open_door"` | Animates named door entity Y position (slide up) | `on_enter` |
| `"checkpoint"` | Saves `player.position` as `game.respawn_point` | `on_enter` |
| `"kill_plane"` | Sets `player.health = 0`, fires `on_player_death` | `on_enter` |
| `"play_sound"` | `Audio(file, autoplay=True, volume=1)` | `on_enter` or `on_exit` |
| `"win_condition"` | Sets `game.state = Game.WIN`, shows Text win screen overlay | `on_enter` |

### Level JSON schema

```json
{
  "type": "trigger",
  "position": [5.0, 0.5, -3.0],
  "scale": [3, 2, 3],
  "on_enter": [
    {"action": "open_door", "target": "door_01"},
    {"action": "play_sound", "file": "assets/sounds/door_open.wav"}
  ],
  "on_exit": []
}
```

`load_level()` creates a `TriggerZone` per entry. The `"target"` field for `"open_door"` resolves by entity name — door entities are placed as named blocks in `level.json` via the v1.2 schema (see [[work/active/v1.2-level-editor-overhaul#Track C — Level Editor JSON Schema Expansion]]).

### Level editor trigger placement

Triggers appear in the editor as semi-transparent box volumes. Selecting a trigger in the hierarchy panel shows trigger-specific inspector fields: scale (x/y/z), on_enter action list (add/remove), on_exit action list. Trigger entities named `trigger_*` — excluded from geometry save logic but included in the `level.json` save.

## System B — Weapon Inventory API (`Scripts/weapon_inventory.py`)

### `WeaponInventory` class

`WeaponInventory` replaces `self.weapon = Weapon(self)` in `Player.__init__`.

```python
class WeaponInventory:
    def __init__(self, player, max_slots=3):
        self.player = player
        self.slots = [None] * max_slots   # weapon instances or None
        self.active_slot = 0

    @property
    def active_weapon(self):
        return self.slots[self.active_slot]

    def give(self, weapon, slot):
        self.slots[slot] = weapon

    def switch_to(self, slot):
        if self.slots[slot] is None:
            return
        if self.active_weapon:
            self.active_weapon.visible = False  # hide, do not destroy
        self.active_slot = slot
        self.active_weapon.visible = True
        self._play_switch_animation()  # 0.2s slide-in

    def next_weapon(self):
        ...  # advance to next occupied slot, wrapping

    def prev_weapon(self):
        ...  # retreat to previous occupied slot, wrapping
```

Key bindings: `1`/`2`/`3` = switch to slot 0/1/2; mouse scroll up/down = `next_weapon()` / `prev_weapon()`; `R` = reload active weapon (double-tap R keeps the magazine, just resets animation).

`switch_to()` hides (not destroys) the current weapon. Weapons are only destroyed when the inventory itself is destroyed — `_clear_gameplay_entities()` calls `inventory.destroy_all()` which destroys weapons in order.

### Weapon types

All three extend the existing `Weapon` base class and use the existing `BulletPool` pattern (see [[brain/Patterns]]).

| Weapon | Damage | Fire rate | Bullet speed | Notes |
|---|---|---|---|---|
| `Pistol` | 25 | 0.15 s | 50 | Refactor of current `Weapon` class |
| `Shotgun` | 15 × 5 pellets | 0.8 s | 30 | ±5° spread; 5 `PlayerBullet` per shot |
| `Rifle` | 40 | 0.08 s | 80 | Hitscan-speed feel; same pool mechanics |

Shotgun pellets share the existing `_player_bullet_pool` — it draws 5 bullets per shot, so the pool size should be reviewed (30 → 50 in v1.5).

### Ammo system

Each weapon carries `ammo: int` and `max_ammo: int`. `ammo = -1` means infinite (Pistol default; Shotgun/Rifle finite). Shooting decrements `ammo`; attempting to fire with `ammo == 0` triggers a dry-click sound and returns without acquiring a bullet. Reload restores `ammo` to `max_ammo` over a weapon-specific animation duration.

HUD ammo counter: a `Text` entity parented to the HUD container (not orphaned to `camera.ui` directly — the HUD container is a named entity in `camera.ui` that gets destroyed with the player). Displays `ammo / max_ammo` next to the health bar.

### `AmmoPickup` entity

`AmmoPickup` extends `AliveEntity`, registered as `Layers.PICKUP` (new bitmask entry — not a damage path). Each frame checks `self.intersects(player).hit`; on true: adds `amount` to the matching weapon type in `player.inventory`; calls `self.die()`. Pickup dies after one collection — pool pattern is not used (pickups are placed by the level, not shot at high frequency).

```json
{"type": "weapon_pickup", "position": [2, 1, -3], "weapon_type": "shotgun"},
{"type": "ammo_pickup",   "position": [3, 1, -3], "weapon_type": "pistol", "amount": 30}
```

A `weapon_pickup` spawns a `Weapon` subclass in the player's inventory the first time the player walks into it. A `ammo_pickup` adds ammo to an already-owned weapon (no effect if the player doesn't own that weapon type).

## Implementation Order

System A and System B are independent — they can be implemented in parallel if two sessions are available, but the order below is safe for a single developer:

**System A:**
1. `Layers.TRIGGER` bitmask entry in `collision_system.py`; update `_swept_blocked` to skip TRIGGER entities

5. `win_condition` action + win screen Text overlay
6. Level editor trigger placement panel (inspector + hierarchy)

**System B:**

8. `Pistol` refactor (current weapon becomes Pistol with ammo)
9. `Shotgun` (multi-bullet shot)
10. `Rifle` (high-speed single bullet)
11. Ammo HUD Text counter
12. `Layers.PICKUP` + `AmmoPickup` entity
13. Level editor weapon/ammo pickup placement

## Constraints

- `Layers.TRIGGER` is not a damage source — never in `COLLISION_MATRIX` as attacker or target
- `Layers.PICKUP` is not a damage source — same rule
- `WeaponInventory.switch_to()` hides the old weapon (sets `visible=False`); it calls `destroy()` only in `destroy_all()` during full scene teardown
- Ammo counter `Text` parented to a named HUD container entity, not orphaned to `camera.ui` (avoids the teardown leak pattern documented in [[brain/Patterns#Explicit teardown at every destroy call site for camera.ui sub-entities]])
- No fourth collision authority — trigger overlap uses `self.intersects()` per frame, not a new raycast system
