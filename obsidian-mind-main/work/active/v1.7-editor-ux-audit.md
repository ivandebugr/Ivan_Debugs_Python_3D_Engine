## Part A — Verified defects (current state, grounded)

Ranked: data loss first, then broken affordances, then chrome.

### D1. Editor open+save strips project textures — silent data loss ⚠
**(probe H4, confirmed.)** Loading the curated level in the editor:
`texture_orange_test.png` arrives as `texture=None` on the block and
`_build_level_data()` re-saves it as `""`. One open+save cycle permanently
blanks every project texture in the file. **This is the root cause of the four
blanked textures in the shipped `level.json`** ([[v1.7-playtest-findings]]
§2.7). Key clue: a *direct* `resolve_texture('texture_orange_test.png')` call
later in the same session returns the texture fine — so resolution works, but
not at `load_existing_level()` time. Probable mechanism: the run started with
no `assets/manifest.json` (it's gitignored; the probe ran in a fresh worktree,
but any manifest rebuild/stale window hits the same path) and the registry
hadn't scanned when load ran. Two independent weaknesses compound:
1. load-time resolution can fail while later resolution succeeds (ordering);
2. **a failed resolve degrades to permanent data loss on save** — the loader
   keeps `None` instead of the original string, and save writes what the
   entity has.
Weakness 2 is the dangerous one: whatever the resolver does, a string that
fails to resolve should survive a round-trip as that same string.

### D2. Sun selection renders the inspector as an overlapping jumble
**(screenshot-confirmed.)** Selecting the sun co-renders the block-only
Texture/Model/Door rows *with* the light section: "Sun Colour" collides with
the Texture row, "Intensity" with "Model", the direction row runs off the
panel bottom, and the Rot X/Y/Z values (`54.204` / `123.690`) overflow their
field boxes into each other. This is exactly the bug class fixed in v1.7
Step 3 for triggers/pickups ("sections must *hide*, not blank" —
[[brain/Gotchas]] inspector entry) — the light type was added afterwards and
never extended those guards (`_update_inspector_texture_swatch` /
`_update_inspector_model_field`'s `has_nonblock`, plus the door row). The
InputField `world_scale` clamp from Step 3 also wasn't applied to the light
fields.

### D3. Toolbar overflows and collides at the default 1280×720
**(screenshot-confirmed.)** At the editor's own default window size the
toolbar reads "…Import Assetizmo: Translate [" — the Gizmo button is clipped
by the screen edge and overlapped by the Import button, and the window's red
exit button (visible in the editor) sits on top of it. The row's total base
width (0.77 + gaps + padding) exceeds the available width even at the 16:9
reference aspect the widths were designed against. Also: the Move button's
`↖` glyph has no definition in the font (`:text(warning): No definition for
character U+2196`) — it renders blank, same family as the Gotchas "no triangle
glyphs" entry.

### D4. Stats strip renders on top of the hint legend
**(screenshot-confirmed.)** `_stats_text` ("entities: 41 colliders: 41") is
placed at `toolbar_left − 0.02` on the same y-band the hint legend occupies;
at 1280×720 they overprint and both become unreadable. The clamp comment in
`_apply_layout` ("doesn't overlap the hint text") doesn't hold in practice.

### D5. Place mode places at the hovered entity's CENTRE, not the cursor
**(probe H1, confirmed for the Place tool; code-read for the ghost.)**
`position = hovered.position + mouse.normal` — clicking anywhere on the
100×100 ground places the block at `(0, 0.5, 0)`, the ground's centre,
regardless of cursor. Fine for unit cubes (centre+normal = adjacent cell),
wrong for any large surface. And the browser **drag-ghost uses the same math**
(`editor_browser.py:981`) — so drag-placement onto the ground has the same
"lands near origin, not under the cursor" behaviour. The v1.2.3 audit
recorded exactly this complaint and its Gotcha prescribes `mouse.world_point`;
the current code in both paths still reads `hovered.position`. Level-building
today survives by placing block-adjacent-to-block (where the math happens to
work) and by inspector coordinate edits.

### D6. The placement preview cube shows in Move mode
**(probe H2, confirmed.)** `update_model_preview()` runs regardless of
`_tool`; hovering any collidable in Move mode shows the translucent white
"this is where a block would go" cube even though a click will select, not
place. Misleading affordance, and it pulses on every hover.

### D7. Typing digits into the sun's inspector fields teleports the camera
**(probe H3, confirmed.)** The bookmark-recall guard in `input()` checks every
typing predicate EXCEPT `_light_typing()` (the light section was added after
that list was written; `_any_field_typing()` and the delete guard both have
it). Typing "3" into intensity/colour recalls camera bookmark 3 mid-edit.
One-line class of fix; listed for completeness because it's data-entry
hostile on the newest feature.

### D8. Save silently drops co-located same-type entities
**(probe H5, confirmed.)** `save_level()` dedups on
`(type, rounded position)`: two blocks at the same position — legitimate for
a rotated pair, a door + frame, or stacked thin plates — serialize as 2
entries and save as 1, silently. No warning, no count in the "Saved level"
message (the entry count printed is post-dedup, so the loss is invisible).

### D9. The spawn marker is an opaque cyan pillar
**(screenshot-confirmed.)** `_spawn_marker` uses
`color.rgba(0, 255, 100, 160)` — 0-255 values into the 0-1 API, the exact
documented clamp footgun ([[brain/Gotchas]] `color.rgb()` entry, CLAUDE.md
convention). Every channel ≥1 clamps: intended translucent green, rendered
**opaque cyan**. It reads as a mystery solid block at the origin (the probe
initially mistook it for a trigger volume, which is the point).

### D10. Browser card labels overlap into unreadability
**(screenshot-confirmed.)** In both tabs, card name labels render wider than
their cards and overprint neighbours ("crosshair_black_crossbaXX_black_…" runs
across four cards; Models tab shows "Cub/Sto/Me/Wo" fragments under overlap).
Compounding: the Textures tab lists engine/UI textures (6× crosshair frames,
blob_shadow, burst) alongside level textures, so the first visible row is
mostly things you'd never place.

Minor observations (no letter, fold into whatever bundle touches that file):
`_finish_box_select`'s exception log prints the *entity* class instead of the
exception class; holding Alt (or Ctrl+Alt) swallows all other editor input for
that keypress via early `return`s; `position_valid()` is pre-existing dead code
(already tracked); `query_near` has no callers.

## Part B — Candidate improvement bundles (cost/risk, no winner)

### E1. Correctness batch — D1 + D2 + D7 + D9  ⭐ highest value-per-effort
Four independent small fixes, all in existing files, no design questions:
- D1: keep the original texture *string* on the entity when resolve fails and
  save that string (never `''`); optionally force a registry scan before
  `load_existing_level()`. ~half day incl. a round-trip test.
- D2: extend the three hide-guards + field `world_scale` clamp to the light
  type, mirroring Step 3's trigger/pickup treatment. ~2–3 hours.
- D7: add `_light_typing()` to the bookmark-recall guard (or replace the
  inline list with `_any_field_typing()`, which exists for exactly this).
  ~15 minutes.
- D9: divide the rgba by 255. ~5 minutes.
- **Risk:** low. D1 needs care that the kept-string path doesn't regress the
  v1.6 `_resolve_texture` routing (item 14 there); test with a missing-file
  name and a present-but-unscanned name.

### E2. Data-safety batch — D8 (+ save messaging)
Options, cheapest first:
1. Log + status-notice when dedup drops entries ("Saved 44 (1 duplicate
   dropped)") — keeps current behaviour, makes it visible. ~1 hour.
2. Widen the dedup key (include scale/rotation/texture) so only true
   duplicates collapse. ~1 hour, small regression risk for levels that today
   *rely* on dedup cleaning up accidental double-placements.
3. Drop dedup entirely and add an explicit "select overlapping duplicates"
   tool later. Cleanest semantics, most behaviour change.
- **Risk:** low for 1, low-medium for 2/3 (changes what old saves would have
  produced).

### E3. Top-bar layout rework — D3 + D4
The toolbar needs measured layout, not tuned constants: either (a) shrink
labels ("Import", "Gizmo [T/R]") and drop the base widths until the row +
stats + hint fit at min supported aspect, ~half day of fiddling; or (b) make
`_apply_layout` wrap the toolbar into two right-anchored rows when width
runs out and give the stats strip its own line, ~1 day, robust at any aspect.
Either way: hide the window exit button in the editor (it overlaps the row
and Ursina already has Esc/Cmd-Q paths) and replace `↖` with an ASCII label
per the existing bracketed-label convention. **Risk:** low — pure
`_apply_layout` bookkeeping, the machinery is solid (v1.2.6 groundwork).

### E4. Place-tool intent — D5 + D6
- Minimal: gate `update_model_preview()` on `_tool == 'place'` (D6) and use
  `mouse.world_point` (snapped) instead of `hovered.position + normal` in all
  three sites — Place-tool click, `update_model_preview`, and the drag ghost
  (`editor_browser.py:981`) — per the v1.2.3 Gotcha's own prescription, so
  every placement path finally agrees. ~2–4 hours + a re-check of the enemy
  `pos[1] += 1` offset against world-point placement.
- Fuller option: fold Place mode into drag-placement entirely (delete the
  tool toggle; the browser drag is the placement affordance people use).
  Bigger UX statement, touches the toolbar and hint text; only worth it if
  Ivan already doesn't use Place mode. Cost ~1 day incl. doc/hint updates.
- **Risk:** low for minimal; medium for the fold (removes a workflow).

### E5. Browser readability — D10
- Truncate card labels to card width with ellipsis (the label Text already
  has per-card width available), ~2–3 hours.
- Optionally: filter engine/UI textures out of the Textures tab via a simple
  prefix/deny-list (`crosshair_*`, `blob_shadow`, `burst`), or a "UI" bucket
  in the manifest scan. ~2–4 hours. Slight risk of hiding something Ivan
  genuinely places; a toggle ("Show all") sidesteps that.

### Explicitly out of scope / unchanged
- **C4/C5 docking**: still parked, still Ivan's call
  ([[v1.7-fix-backlog]] item 2). Nothing above requires or blocks it.
- **Hierarchy naming** (rows are "B (3.0,1.5,8.0)" coordinates): readable
  enough at this level size; a naming scheme is a feature, not a defect —
  raise only if the level count grows.
- The editor-light-tooling harness scenario (fix-backlog item 5) still wants
  its committed home; `tools/editor_ux_probe.py` covers some of that ground
  in the meantime.
