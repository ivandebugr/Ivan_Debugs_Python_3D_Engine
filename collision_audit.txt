How many separate expiry checks exist?

    No competing mechanism. ✓

  EnemyBullet:
    ONE check: enemy.py lines 40-41 — distance from world origin > 500.
    No spawn_time / MAX_LIFETIME check. This is a distance-based expiry, not
    time-based. It works but is inconsistent with PlayerBullet's approach and

    slowly near origin. Low severity but should be unified.

Can destroy() be called twice on the same bullet?

  PlayerBullet: LOW RISK within its own update() — every _destroy() path has
    a return immediately after. But if _destroy() is called while the entity
    is still in scene.entities for the current frame tick, there is no guard
    flag preventing a second call from a future frame before Ursina removes it.
    In practice this is safe because _destroy() calls destroy(self) and Ursina
    defers removal — subsequent frames won't call update() on a destroyed entity
    once Ursina processes the destroy queue. No observed double-destroy bug here.

  EnemyBullet: HIGH RISK.
    Path 1 (Entity.update runs BEFORE global update in Ursina):
      EnemyBullet.update() fires → raycast hits player → damage applied →
      destroy(self) called → return.
    Path 2 (global update in main.py runs immediately after):
      main.py lines 230-234 iterate scene.entities looking for EnemyBullet.
      Ursina defers destroy to end of frame; the bullet is still in
      scene.entities during the global update() call.
      bullet.intersects(player) → True → player.health -= 20 AGAIN →
      destroy(bullet) AGAIN.
    Result: player takes double damage per EnemyBullet hit; destroy() called
    twice on the same bullet. CONFIRMED BUG.
